- **Findings**:
  - **DCR is the canonical superset/upstream** (strict superset of llm's shared infra; 18 vs 17 tools, 9 vs 4 generic-relevant skills; llm has zero generic tools/skills DCR lacks). CLAUDE.md "not git-initialized" claim is STALE — all 4 are git repos on main.

Coupling via $CLAUDE_PROJECT_DIR/viewer/tools + $CLAUDE_PROJECT_DIR/surveys in hooks.
  - **Per-repo (must NOT unify)**: settings.json, surveys/ content, repo-specific skills, localized docstrings (deliberate per decision 2026-06-17-01), bare-refs-severity, download/, viewer.content.json, CLAUDE.md/MEMORY.

## Conversation 2
- **Request**: "which one is the best and why" (user interrupted the AskUserQuestion menu, wants a direct recommendation).
- **Actions**: Gave a decisive recommendation with reasoning grounded in the drift map.
- **Result**: Recommended **A — formalize the sync** (committed, drift-checked vendoring script; upstream = DCR) over packages/symlinks/hybrid. Concrete shape: `survey-kit.manifest.json` (upstream, shared file set, per-repo localization substitutions + exclusions) + `sync-kit.py` with `--check` (drift gate, wire into /check-survey) and `--apply`. Offered next step: full brainstorm→spec→plan vs. sketch-and-iterate.
- **Findings**: Reasoning hinges on 4 facts — (1) the pain is source *drift*/manual porting, not duplication; (2) ~95% of drift is deliberate localization that packages/symlinks can't express but a per-repo substitution map can (copy upstream logic, re-apply local examples); (3) sync is the only mechanism that uniformly covers the whole stack incl. the per-repo subsets (sionna .claude/tools, prach 3-skills, llm dropped telecom commands) — packaging would *raise* the mechanism count; (4) keeps repos self-contained/clone-safe (DCR needed only at pull time). node_modules (~200MB) dedup explicitly de-prioritized: regenerable git-ignored artifact, not version-controlled spread. Escalate to packaging only if global `survey-*` CLI or DCR-less installs are wanted.
## Conversation 3
- **Request**: "but how will I sync the updates in the siblings to the upstream?" (reverse-direction sync — the hard part of copy-vendoring).
- **Actions**: Explained the siblings→upstream story; owned that reverse sync is the weak point of any vendoring scheme.
- **Result**: Answer = three-bucket classification in the manifest (pull-only / shared-bidirectional / local-only) + author-upstream policy to keep up-flow rare + `sync-kit.py --promote <path>` escape hatch that de-localizes a sibling file (reverse substitution-map + path remap) and emits a logic-only patch to land in DCR via format-patch/apply. The same substitution map powers pull-down, push-up, AND the `--check` drift classifier (localization-only vs logic drift). Structural fix that makes promote trivial: externalize examples so shared files are byte-identical across repos (also the packaging prerequisite). Offered to draft the manifest seeded from the drift map.
- **Findings**: A lot of the "this-repo-ahead" drift is local-only (sionna kernel cross-checks, llm method taxonomy) and must NOT be promoted — only the small shared-bidirectional bucket flows up. Genuine stranded-generic downstream artifacts identified: llm's enrich-equation.md command + survey-explainer-fold "how-large" mode. Cross-repo promote is format-patch/apply (no shared git history), not cherry-pick.
## Conversation 4
- **Request**: "is this scheme too complicated?"
- **Actions**: Self-corrected — agreed the back half is over-built; re-scoped to a right-sized design.
- **Result**: Conceded the substitution maps / `--promote` / de-localize / externalize-examples refactor are YAGNI for a low-frequency, ~2-active-repo, single-dev setup — they existed only to auto-preserve cosmetic docstring localization. Right-sized design: an **upstream-managed path list** + `sync.sh --check` (`diff -rq` against DCR, wired into /check-survey) + `--pull` (rsync down). Reverse sync = manual + author-in-DCR policy. Optional simplifier: neutralize docstrings so shared files are byte-identical and "locally owned" shrinks to genuinely-divergent files. Principle stated: **automate detection (silent drift), not reconciliation (rare, humans fine at it).** Offered to write the ~50-line version.
- **Findings**: The earlier elaborate proposal over-automated the part humans are fine at; the only real value-add is making drift *visible*. Complexity was driven by an unexamined requirement (auto-preserve docstring examples) that should be dropped, not engineered around.
## Conversation 5
- **Request**: "Can you make it even simpler for me? The mechanism could be complex and as stable as possible, but human effort should be minimized."
- **Actions**: Recognized the objective flip — minimize *human effort* + maximize stability, not minimize code. Pivoted from sync-script (copies + `--pull`) to **reference-not-copy via Claude Code user scope**.
- **Result**: New recommendation = stop copying the stack into each repo; hoist the generic stack to ONE source of truth so there is nothing to sync. (a) generic skills/commands/rules/validation-hooks → `~/.claude/` (user scope, auto-applies to all projects); (b) viewer + Python toolchain → one canonical dir (e.g. `~/.claude/survey-kit/`, its own git repo) referenced by the user-scope hooks + a `survey-view` PATH wrapper (serve.js already resolves content from cwd); (c) each repo's `.claude/` keeps only repo-specific overrides + settings + surveys/. Drift/pull/push/reverse-sync cease to exist by construction. Tradeoffs (set-once): fresh machine needs `git clone … ~/.claude/survey-kit` one-line bootstrap; user-scope hooks get a `[ -d "$CLAUDE_PROJECT_DIR/surveys" ] || exit 0` guard so they stay silent in the ~60 non-survey repos. Offered to lay out the exact hoist + hook re-wiring before touching anything.
- **Findings**: The duplication problem dissolves if the shared infra lives where Claude Code already shares across repos (user scope) + a single referenced kit dir; project-skill-overrides-user-skill gives clean generic/domain layering; prach's staleness self-heals once it stops carrying its own copy. Saved feedback memory: user prefers decisive right-sized recs + minimal human effort.
## Conversation 6
- **Request**: "but I truly want the skills repo-wide, not global at all" — rejects the `~/.claude/` user-scope hoist; wants skills repo-local (in each repo's `.claude/`, travels with the repo).
- **Actions**: Pivoted back to in-repo copies, but optimized the propagation for minimal human effort + max stability (per Conv 5 objective).
- **Result**: Recommendation = **canonical kit repo (single source of truth) + each repo keeps its own committed copy + one-command propagation (`kit pull` / `kit pull-all`) + `kit check` wired into /check-survey so drift is auto-flagged**. Stability key: keep the shared set BYTE-IDENTICAL across repos (drop the cosmetic Vaswani-vs-3GPP docstring tuning) so propagation is a dumb overwrite — no merges, no substitution transforms to misfire. Domain-specific skills stay as repo-only skills outside the shared set, never touched by pull. Reverse flow = edit kit, pull-all (rare). Explicitly rejected (given "as stable as possible"): an auto-overwrite daemon (can clobber mid-edit) and `git subtree` (merges → conflicts + per-repo commands; dumb-overwrite is more stable and does all repos at once). One open decision surfaced: whether any *shared* skill needs genuine per-repo domain text (→ becomes a hand-owned overlay excluded from sync); recommended keeping shared skills identical + domain specialization in separate repo-only skills.
- **Findings**: "repo-wide not global" = repo scope (in-repo, self-contained, pushable), confirming copies must live in each repo; minimal-effort within that constraint = single canonical kit + dumb-overwrite propagation + drift surfaced through the existing gate, NOT fully-automatic propagation (sacrifices the stability priority).
