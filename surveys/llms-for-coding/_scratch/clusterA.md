### Q4: pass@k metric — unbiased estimator, n≥k rationale, functional correctness vs BLEU

- **Findings**:
  - **Unbiased estimator (Codex §2.1, Eq. (1)):** pass@k := E_Problems[ 1 − ( C(n−c, k) / C(n, k) ) ], where C(·,·) is the binomial coefficient. n = total samples per task, c = number of correct samples (c ≤ n) that pass unit tests, k = the k in pass@k.
  - **n ≥ k sampling rationale (Codex §2.1):** The naive estimator — "generate k samples per task, problem solved if any passes, report fraction solved" — has *high variance*. Instead they generate n ≥ k samples per task (paper uses **n = 200, k ≤ 100**), count correct c ≤ n, and compute the unbiased estimator above. Direct evaluation of the estimator "results in very large numbers and numerical instability"; they give a numerically stable numpy implementation (Codex Fig. 3): `def pass_at_k(n, c, k): if n - c < k: return 1.0; return 1.0 - np.prod(1.0 - k / np.arange(n - c + 1, n + 1))`.
  - **Why not 1−(1−p̂)^k:** "One may be tempted to estimate pass@k with 1−(1−p̂)^k where p̂ is the empirical estimate of pass@1, but we show that it is biased in Appendix A." (Codex §2.1).
  - **Functional correctness vs match-based metrics (Codex §2.1):** Generative code models are "predominantly benchmarked by matching samples against a reference solution, where the match can be exact or fuzzy (as in BLEU score)." Match-based metrics are deficient: Ren et al. (2020) finds BLEU "has problems capturing semantic features specific to code"; more fundamentally, match-based metrics "are unable to account for the large and complex space of programs functionally equivalent to a reference solution." A sample is judged *correct* iff it passes a set of unit tests. Codex also shows empirically that "functionally inequivalent programs generated by our model … often have higher BLEU scores than functionally equivalent ones" (Codex §2.1 / Appendix). "The most convincing reason to evaluate functional correctness is that it is used by human developers to judge code" (test-driven development framework).
  - Temperature for sampling experiments: 0.8.
- **Sources**: Chen et al., "Evaluating Large Language Models Trained on Code" (Codex), 2021 — (local: download/chen-codex-evaluating-llms-code-2021.pdf), §2.1, Eq. (1), Fig. 3.
- **Confidence**: high
- **Gaps**: Appendix A bias-proof not transcribed (only the claim that 1−(1−p̂)^k is biased); not needed for survey orientation.

### Q2: Historical lineage — encoder (CodeBERT) → decoder/Codex → open code models (CodeGen, InCoder, Code Llama)

Per-milestone facts (year, sizes, training-data scale, headline HumanEval) pulled directly from each paper:

**CodeBERT (Feng et al., 2020; arXiv:2002.08155, v4 dated 18 Sep 2020).** *Architecture:* encoder-only, "exactly the same model architecture as RoBERTa-base," **125M parameters** (CodeBERT §3.1). *Bimodal* PL+NL model. *Objective:* hybrid — masked language modeling (MLM) + **replaced token detection (RTD)** (detect plausible alternatives sampled from generators), letting it use both bimodal NL-PL pairs and unimodal code. *Training data:* CodeSearchNet corpus (Husain et al. 2019) — **2.1M bimodal datapoints + 6.4M unimodal codes** across **6 languages** (Python, Java, JavaScript, PHP, Ruby, Go). *Evaluation:* NL code search (MRR on CodeSearchNet), code documentation generation, NL-PL probing — **NOT HumanEval pass@k** (it is not a generative completion decoder, so no pass@1 is reported). *What it unlocked:* general-purpose pretrained PL+NL representations for understanding/retrieval downstream tasks.

**Codex (Chen et al., 2021; arXiv referenced in repo file).** *Architecture:* GPT decoder fine-tuned from GPT-3 family for code generation. *Training data:* collected May 2020 from **54M public GitHub repos**, **179 GB unique Python files <1 MB**, filtered to a **final 159 GB** dataset (Codex §3.1). *Headline HumanEval (164 hand-written problems, 7.7 tests avg):* Codex-12B solves **28.8% (pass@1)**; GPT-3 0%, GPT-J 11.4% (Codex Abstract/§1). With 100 samples/problem, **70.2%** of problems solved (any sample passes); selecting best by mean log-prob → 44.5%, oracle (passes tests) → **77.5%**; temperature 0.8 (Codex Fig.1/§1). Codex-S (further fine-tuned on standalone functions) → **37.7% pass@1**. *What it unlocked:* large-scale functional-correctness code generation from docstrings; powers GitHub Copilot; introduced the HumanEval benchmark + pass@k metric.

**CodeGen (Nijkamp et al., 2022; arXiv:2203.13474; "Published as a conference paper at ICLR 2023").** *Architecture:* standard transformer autoregressive LM family at **350M, 2.7B, 6.1B, 16.1B** params; open-sourced JAXFORMER training library (TPU-v4). *Training data:* trained sequentially on **THEPILE** (825.18 GiB English text, Gao et al. 2020), **BIGQUERY** (multi-lang code), then **BIGPYTHON** (Python) → NL / MULTI / MONO variants (CodeGen §2.1). *Headline HumanEval (CodeGen Table 1, Appendix G):* CODEGEN-MONO 16.1B = **pass@1 29.28%**, pass@10 49.86%, pass@100 75.00%; MONO 6.1B = 26.13/42.29/65.82; NL 16.1B = 10.92/38.43/62.76. *Also:* MTPB — Multi-Turn Programming Benchmark, 115 problem sets factorized into multi-turn prompts; multi-turn factorization significantly improves synthesis. *What it unlocked:* first fully-open competitive code LLM family + multi-turn program-synthesis paradigm.

**InCoder (Fried et al., 2022; arXiv:2204.05999; ICLR 2023).** *Architecture:* decoder generative model, **6.7B and 1.3B** params (InCoder-6.7B is the flagship). *Objective:* **causal masking (CM)** (Aghajanyan et al. 2022a) — first large generative code model able to infill arbitrary regions with bidirectional context (see Q3). *Training data:* permissively-licensed code, **final corpus 159 GB code** (52 GB Python, plus 57 GB StackOverflow content) (InCoder §2.1/A.1). *Headline HumanEval:* InCoder-6.7B = **pass@1 15.2%**, pass@10 27.8%, pass@100 47.0% (InCoder Table, left-to-right synthesis). *What it unlocked:* zero-shot code infilling (type inference, comment generation, variable renaming) — bidirectional context "substantially improves" infill tasks while staying competitive on L2R synthesis.

**Code Llama (Rozière et al., 2023; arXiv:2308.12950).** *Architecture:* family built on Llama 2 — foundation (Code Llama), Python specialization (Code Llama - Python), instruct (Code Llama - Instruct), each at **7B, 13B, 34B, 70B**. *Context:* trained on **16k-token** sequences; additional long-context fine-tune extends RoPE from 4,096 → **100,000 tokens** (Code Llama §abstract, §2). Specialization pipeline (Fig. 2) token budgets: Llama 2 foundation → +**500B** code tokens → Python +100B/+20B → long-context +20B → instruct +5B. *Infilling:* 7B/13B/70B support FIM (PSM+SPM, Bavarian-style — see Q3). *Headline HumanEval pass@1 (Code Llama Table 2, greedy decoding):* **foundation** Code Llama 7B = 33.5%, 13B = 36.0%, 34B = 48.8%, 70B = 53.0%; **Code Llama - Python** 7B = 38.4%, 13B = 43.3%, 34B = 53.7%, **70B = 57.3%**; **Code Llama - Instruct** 70B = **67.8%** (the abstract's "up to 67%" top-line). MBPP pass@1 up to 65.6% (Python 70B). For reference in the same Table 2: GPT-4 67.0%, code-cushman-001 (12B) 33.5%, StarCoder-Python (15.5B) 33.6%, Llama 2 70B 30.5%. *What it unlocked:* open SOTA code models with infilling + long context + instruction following.

  - Chen et al., "Evaluating LLMs Trained on Code" (Codex), 2021 — (local: download/chen-codex-evaluating-llms-code-2021.pdf), §1, §3.1.
  - Nijkamp et al., "CodeGen," 2022/ICLR2023 — (local: download/codegen-2022.pdf), §2.1, Table 1 + Appendix G.
  - Fried et al., "InCoder," 2022/ICLR2023 — (local: download/incoder-2022.pdf), §2.1, HumanEval table.

### Q3: Fill-in-the-middle (FIM) transform — PSM/SPM construction, sentinels, why code needs infilling

**Why code needs infilling (motivation).** "Code is seldom written in a single left-to-right pass and is instead repeatedly edited and refined" (InCoder Abstract). A purely causal (left-to-right) LM "only condition[s] on context to the left of the generated tokens, thus preventing infilling"; a masked LM (BERT) can use both left and right context but "their training objective is typically limited to generating only about 15% of a document" (InCoder §2). FIM/causal-masking combines both: autoregressive generation of full documents AND infilling conditioned on **both left (prefix) and right (suffix) context** — essential for code where an edit site has committed code on both sides.

**Bavarian et al. PSM transform (exact document rewriting).** The core transformation, applied to a fraction of the dataset: split each document into three pieces at random and move the middle to the end (Bavarian §2/§3):
  `document → (prefix, middle, suffix) → (prefix, suffix, middle)`
Pieces are split at the **character level**, splitting locations sampled independently from a uniform distribution over document length (Code Llama §2.2 restating Bavarian). The three sections are concatenated with sentinel tokens **`<PRE>`, `<MID>`, `<SUF>`**. The PSM (prefix-suffix-middle) tokenized document is exactly:
  **`<PRE> ∘ Enc(prefix) ∘ <SUF> ∘ Enc(suffix) ∘ <MID> ∘ Enc(middle)`**  (Bavarian §3, "PSM"), where `∘` is concatenation. Documents (FIM or plain AR) are joined with `<EOT>`. Loss is kept on all three sections (model learns to generate prefix, suffix, AND the middle).

**SPM mode (suffix-prefix-middle).** Ordering `[suffix, prefix, middle]` (Bavarian §3.1, App. D). A "natural" encoding (SPM variant 1) would be `<SUF> ∘ Enc(suffix) ∘ <PRE> ∘ Enc(prefix) ∘ <MID> ∘ Enc(middle) ∘ <EOT>`, but the encoding actually used (**SPM variant 2**) is:
  **`<PRE> ∘ <SUF> ∘ Enc(suffix) ∘ <MID> ∘ Enc(prefix) ∘ Enc(middle) ∘ <EOT>`**
The variant-2 reason: it makes SPM data "occur naturally as part of PSM training" (when a document is split uniformly), giving positive transfer between the two formats rather than separating them. *Pseudocode (Bavarian §3):* `character_level_psm_fim` returns `[sentinel("prefix"), *prefix, sentinel("suffix"), *suffix, sentinel("middle"), *middle]`.

**Span selection / implementation choices.** Spans split at **character level** (recommended) so the model produces sensible completions even when prefix/suffix end mid-token (Bavarian §8.1). Two pipeline placements: **document-level** vs **context-level** FIM (context-level is superior; document-level is a simpler option). InCoder's variant: number of masked spans drawn from a Poisson(mean=1) truncated to [1,256] (≈single span ~50% of the time), span endpoints uniform, overlapping spans rejected; each span replaced by sentinel `<Mask:k>`, moved to document end prepended with its sentinel and appended with `<EOM>` (InCoder §2.1).

**FIM rate / "FIM-for-free."** A **FIM rate of 50%** incurs no performance hit in left-to-right capabilities (same AR loss, lower FIM loss) — "FIM models can be considered strictly better than AR models" (Bavarian §2, §4.2, Fig. 1/3). Models see original data only 50% of the time yet learn the new capability for free. *Recommendations (Bavarian §8.1):* apply FIM at character level, include character-level random spans for mid-token robustness; PSM inference can beat SPM for mid-token robustness, but **joint PSM+SPM pretraining yields the best performance** (positive transfer); context-level FIM superior, document-level acceptable. Code Llama uses exactly this: half the splits PSM, half SPM, FIM applied with probability 0.9 (Code Llama §2.2), extending Llama 2's tokenizer with 4 special tokens (prefix/middle/suffix begin + ...).

  - Fried et al., "InCoder," 2022 — (local: download/incoder-2022.pdf), §2, §2.1 (causal-masking construction, sentinel `<Mask:k>`/`<EOM>`).
  - Rozière et al., "Code Llama," 2023 — (local: download/code-llama-2023.pdf), §2.2 (PSM/SPM 50/50, char-level, p=0.9, 4 sentinel tokens — cites Bavarian 2022 App. D).

### Q5: Autoregressive code LM basics — next-token objective and code tokenization intuition

- **Findings** (orientation-level):
  - **Next-token objective on code.** Code LMs are "standard transformer-based autoregressive language models" trained to maximize the likelihood of a corpus of code (CodeGen §2.1; InCoder §2 "trained to maximize the likelihood of a corpus of code"). The objective is the ordinary left-to-right next-token prediction p(x_t | x_<t) applied to token sequences that happen to be source code; Codex itself is GPT-3-family fine-tuned on code (Codex §3.2). No code-specific loss is needed for the base capability — FIM/causal-masking (Q3) is an *added* objective layered on top.
  - **Tokenization intuition — whitespace/indentation problem.** Naive byte/BPE tokenizers waste tokens on code's heavy repeated whitespace and indentation. CodeGen handles this by **extending the GPT-2 BPE vocabulary with special tokens representing repeating runs of tabs and whitespaces** (CodeGen §2.1, preprocessing step 3). This compresses Python's significant indentation.
  - **Byte-level BPE + cross-whitespace merges.** InCoder trains a **byte-level BPE tokenizer** (Sennrich et al. 2016; Radford et al. 2019 / GPT-2 lineage) and crucially **allows tokens to extend across whitespace (excluding newlines)** so common code idioms like `import numpy as np` become *single* tokens. This "substantially improves the tokenizer's efficiency — reducing the total number of tokens required to encode our training corpus by **45%** relative to the byte-level BPE tokenizer and vocabulary of GPT-2" (InCoder §2.1). Byte-level fallback also guarantees no out-of-vocabulary tokens for arbitrary identifiers/Unicode.
  - **SentencePiece nuance (Code Llama).** Code Llama (SentencePiece tokenizer, Kudo & Richardson 2018) suppresses the implicit leading space SentencePiece prepends, to limit AR↔infilling distribution shift (Code Llama §2.2).
- **Sources**:
  - Nijkamp et al., "CodeGen," 2022 — (local: download/codegen-2022.pdf), §2.1 (GPT-2 BPE + tab/whitespace special tokens; AR transformer).
  - Fried et al., "InCoder," 2022 — (local: download/incoder-2022.pdf), §2.1 (byte-level BPE, cross-whitespace merges, 45% token reduction).
  - Chen et al., "Codex," 2021 — (local: download/chen-codex-evaluating-llms-code-2021.pdf), §3.2 (GPT-family AR fine-tune).
  - Rozière et al., "Code Llama," 2023 — (local: download/code-llama-2023.pdf), §2.2 (SentencePiece leading-space suppression).
- **Confidence**: high
- **Gaps**: Exact vocabulary sizes per model not transcribed (orientation only, not requested). General "next-token / cross-entropy" formula is standard and can be stated from any transformer reference; not a code-specific citation.

### Q1: Why code is a distinct modality + taxonomy of code tasks

- **Findings**:
  - **Formal syntax + executability.** Unlike NL, code has a formal grammar and is *executable* — it can be run against unit tests. Codex operationalizes this: "a sample is considered correct if it passes a set of unit tests," and it builds a gVisor/Kubernetes sandbox to safely execute untrusted generated programs (Codex §2.1, §2.3). Executability is what makes the next point possible.
  - **Functional correctness = verifiable ground truth.** Code uniquely admits an objective, automatable correctness signal: pass/fail on tests. Codex argues match-based metrics (exact match, BLEU) are deficient for code because they "are unable to account for the large and complex space of programs functionally equivalent to a reference solution," and shows functionally *inequivalent* programs can score *higher* BLEU than equivalent ones (Codex §2.1). This is the central reason code is a distinct modality vs NL: the ground truth is *behavioral*, not lexical. (Cross-ref Q4.)
  - **Long-range / bidirectional structure.** Code edits commit context on *both* sides of an insertion point — "code is seldom written in a single left-to-right pass and is instead repeatedly edited and refined" (InCoder Abstract) — motivating infilling with bidirectional context (Q3). Code also has long-range dependencies (a function defined far above its call site); Code Llama explicitly targets this with 16k-token training and 100k-token long-context extension (Code Llama §abstract, §2).
  - **Distinctive tokenization pressure.** Significant whitespace/indentation and long identifiers make code tokenization a distinct problem (see Q5 — tab/whitespace special tokens, cross-whitespace BPE merges).
  - **Taxonomy of code tasks** (each grounded in an acquired source):
    - *Completion* (left-to-right continuation) — standard AR generation (CodeGen, Codex).
    - *NL→code synthesis* (docstring/spec → function) — Codex HumanEval, CodeGen "program synthesis from NL descriptions" (CodeGen Abstract); multi-turn synthesis (CodeGen MTPB).
    - *Infilling / fill-in-the-middle* (generate a span given left+right context) — InCoder, Bavarian FIM, Code Llama (Q3).
    - *Program repair / editing* — InCoder frames editing "via masking and infilling" (InCoder Abstract).
    - *Code translation* (one language → another) — Codex §2.1 cites unsupervised code translation (Lachaux et al. 2020) as a motivating functional-correctness task.
    - *Test generation* — implied by the unit-test-centric evaluation paradigm (Codex); not a headline contribution of any cluster-A paper (gap below).
    - *Repo-level / multi-file editing* — beyond single-function scope; foreshadowed by Code Llama's long-context (100k tokens) but not benchmarked in cluster-A papers (gap below).
    - *Code understanding / retrieval* (non-generative) — CodeBERT: NL code search, documentation generation, type inference, variable renaming, comment generation (CodeBERT Abstract; InCoder §4 infill-based versions of the latter three).

  - Fried et al., "InCoder," 2022 — (local: download/incoder-2022.pdf), Abstract, §4 (editing, bidirectional context, infill tasks: type inference / comment gen / variable rename).
  - Nijkamp et al., "CodeGen," 2022 — (local: download/codegen-2022.pdf), Abstract (NL→code synthesis, multi-turn).

- **Confidence**: high for completion/synthesis/infilling/repair/translation/understanding (each anchored in an acquired full-text source); medium for the *test-generation* and *repo-level editing* task categories — they belong in the taxonomy but are not headline contributions of any cluster-A paper.
