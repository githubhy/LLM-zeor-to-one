# Cluster D evidence — §10 Inference/Decoding/Serving, §11 Retrieval/Repo Context, §12 Agentic Coding

Phase-3 evidence collection. Findings carry locators; sources carry source tags.

### Q1: FIM inference (prompting an FIM-trained model at completion) + constrained/grammar-guided decoding (PICARD, grammar-constrained)

- **Findings — FIM inference (Bavarian et al. 2022, "Efficient Training of LMs to Fill in the Middle")**:
  - FIM training transform (§3, "FIM training and inference"): a document is split into (prefix, middle, suffix), the middle span is moved to the end, and the three sections are concatenated with sentinel tokens `<PRE>`, `<SUF>`, `<MID>`. The tokenized FIM document is `<PRE> ∘ Enc(prefix) ∘ <SUF> ∘ Enc(suffix) ∘ <MID> ∘ Enc(middle)` (this is the **PSM** = prefix-suffix-middle order). Sentinel-token concatenation follows Donahue et al. 2020, Aghajanyan et al. 2022, Fried et al. 2022 (InCoder).
  - **Inference / how the model is prompted at completion time (§3, PSM inference)**: "For inference, we encode the given prefix and suffix and prompt the model with `<PRE> ∘ Enc(prefix) ∘ <SUF> ∘ Enc(suffix) ∘ <MID>`." The model then *continues sampling* the middle span until it emits an `<EOT>` token, which is how it signals it has joined the prefix to the suffix. If it fails to emit `<EOT>` the join is unsuccessful.
  - **SPM variant (§3.1)**: swaps order to suffix-prefix-middle. Motivation is improved **key-value caching during inference** — with SPM, appending tokens to the prefix no longer invalidates the K/V already computed for the suffix section (advantage is workload-dependent; minor suffix changes still invalidate prefix cache).
  - **FIM-for-free property (abstract / Fig. 3 / §1)**: jointly training with a 50% FIM rate "incurs no cost" — the left-to-right autoregressive loss trend is unchanged even though the model sees original-order data only 50% of the time, so it gains infilling capability for free.
- **Findings — PICARD constrained decoding (Scholak et al. 2021)**:
  - Mechanism (§2): incremental parsing during beam search — at each decoding step PICARD **rejects inadmissible tokens** by checking the detokenized partial output. Four comprehensiveness modes (§2): `off` (no checking), `lexing`, `parsing without guards` (grammatical / AST level), and `parsing with guards` (highest — also detects keyword spelling errors and rejects table/column names invalid for the given SQL schema). A prediction passing a higher mode always passes lower modes. Only the top-k (top-2/-4/-8) token predictions per hypothesis are checked in beam search; rest auto-discarded.
  - **Effect on validity**: without PICARD, ~**12%** of T5-3B's generated SQL queries on Spider dev fail with an execution error; with PICARD only ~**2%** of predictions are unusable (beam search exited without a valid SQL).
  - **Effect on accuracy (Spider, T5-3B + PICARD, Table 1)**: lifts T5-3B to SOTA — exact-set-match **75.5%** dev / **71.9%** test; execution accuracy **79.3%** dev / **75.1%** test; test-suite execution accuracy 71.9% (test). On par with / higher than closest competitor LGESQL+ELECTRA. Also evaluated on CoSQL.
- **Sources**:
  - Bavarian, Jun, Tezak, Schulman, McLeavey, Tworek, Chen et al., "Efficient Training of Language Models to Fill in the Middle," 2022 — (local: download/fim-bavarian-2022.pdf)
  - Scholak, Schucher, Bahdanau, "PICARD: Parsing Incrementally for Constrained Auto-Regressive Decoding from Language Models," EMNLP 2021 — (local: download/picard-2021.pdf)
- **Confidence**: high
- **Gaps**: General grammar-constrained decoding (GCD / constrained semantic parsing beyond SQL, e.g. EBNF/CFG-masked decoding, Outlines/llguidance-style libraries) is not covered by an acquired primary source — only the SQL-specific PICARD instance. Flag for a general-GCD citation if §10 needs the broader claim.

### Q3: Repository-level context — RepoCoder (iterative retrieve-generate), code retrieval/embeddings, long-context-vs-retrieval tradeoff

- **Findings — RepoCoder (Zhang et al. 2023, "RepoCoder: Repository-Level Code Completion Through Iterative Retrieval and Generation")**:
  - **Method (abstract / §3)**: a *simple, generic* framework pairing a similarity-based retriever with a pre-trained code LM in an **iterative retrieve-generate pipeline**. Retrieval DB built by a **sliding window** over repo files (window size `S_w`, fixed stride per step). Default retriever is a **sparse** one: code snippets are bag-of-tokens and similarity is the **Jaccard index** `|S_q ∩ S_c| / |S_q ∪ S_c|`. A dense retriever based on UniXcoder was also tried (Appendix B) and the simple sparse retriever achieves equivalent performance — highlighting robustness.
  - **Iterative mechanism (§3)**: the key idea — at iteration 1 the unfinished code `X` is the retrieval query, but it is a poor proxy for the intended completion. At the i-th iteration (i>1) RepoCoder **augments the retrieval query with the previously generated completion** `Ŷ^{i-1}`: `C^i_ret = R(C_repo, X, Ŷ^{i-1})`, then builds a new prompt from those snippets. This bridges the gap between retrieval context and completion target. Two iterations give higher recall for ground-truth API invocations than one (contributes to gains); choosing the optimal iteration count is noted as an open challenge (subsequent iterations can be unstable, Appendix D).
  - **Benchmark**: introduces **RepoEval**, built from latest high-quality GitHub repos, covering **line / API-invocation / function-body** completion. Line scenario: 8 repos, 200 random lines each. Metrics: Exact Match (EM, binary) and Edit Similarity (ES).
  - **Gains (locators)**: "significantly improves the In-File completion baseline by **over 10% in all settings**" and "consistently outperforms the vanilla retrieval-augmented (RAG, single-shot) approach." Absolute improvements: **EM exceeds 10% and ES exceeds 8%** over the In-File baseline. Evaluated with four LMs of varying size including GPT-3.5-Turbo and CodeGen. Comparison points include an **oracle** single-retrieval method (uses ground-truth-adjacent context) — RepoCoder approaches/beats it.
- **Findings — long-context vs retrieval tradeoff**: RepoCoder's own framing motivates retrieval because relevant info is "scattered in different files" and the unfinished code is a weak query — it does not run a head-to-head long-context-window vs retrieval ablation. The long-context-vs-retrieval comparison is therefore NOT quantified by an acquired primary source here.
- **Sources**:
  - F. Zhang, B. Chen, Y. Zhang, J. Keung, J. Liu, D. Zan, Y. Mao, J.-G. Lou, W. Chen, "RepoCoder: Repository-Level Code Completion Through Iterative Retrieval and Generation," EMNLP 2023 — (local: download/repocoder-2023.pdf)
- **Confidence**: high (RepoCoder method + gains); low for the long-context-vs-retrieval tradeoff (no acquired source)
- **Gaps**: No acquired primary source for (a) the long-context-window vs RAG head-to-head for whole-repo context, (b) code-specific embedding/retriever models as a standalone topic (CodeBERT is in download/ for §earlier clusters; UniXcoder, CodeT5 retrievers not acquired). Flag if §11 needs a long-context-vs-retrieval citation.

### Q4: Agent-Computer Interface — SWE-agent ACI design + ReAct (reason+act interleaving) + SWE-bench resolved-% numbers

- **Findings — SWE-agent ACI (Yang, Jimenez, Wettig, Lieret, Yao, Narasimhan, Press 2024, "SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering")**:
  - **Core thesis**: LM agents are a new category of end user; a purpose-built **agent-computer interface (ACI)** — the set of commands the agent uses and the format of feedback it receives — substantially improves agent performance over human-oriented UIs (the raw Linux shell). The ACI has principal components: search/navigation, file viewer, file editor, context management. At each step the agent emits a thought + a command and incorporates the command's execution feedback (explicitly **ReAct-style**, citing Yao et al.).
  - **Four ACI design principles (§ intro)**: (1) Actions should be **simple and easy to understand** for agents (few options, concise docs — vs. dozens of bash flags); (2) Actions should be **compact and efficient** (consolidate file navigation/editing into as few actions as possible so one step makes real progress); (3) Environment **feedback should be informative but concise** (substantive state info without clutter, e.g. show revised content after an edit); (4) **Guardrails mitigate error propagation and hasten recovery** (e.g. a code syntax checker / linter integrated into the edit action that flags mistakes for quick correction).
  - **SWE-bench resolved-% (headline)**: with **GPT-4 Turbo** as base LM, SWE-agent resolves **12.47% (286/2,294)** of the full SWE-bench test set and **18.00% (54/300)** of the SWE-bench **Lite** split. Abstract rounds the full-set pass@1 to **12.5%**. Also **87.7%** pass@1 on **HumanEvalFix**. Portable across LMs: with **Claude 3 Opus**, **10.5%** of the benchmark. Prior best (non-interactive RAG system) was **3.8%** on the full set — so SWE-agent is ~3.3x better on full test and reports a **6.7-fold** higher %Resolved than RAG on Lite (at 8–13x the cost).
  - **ACI ablation (§5.1, on 300-instance Lite, base 18.0)**: removing the custom editor → 10.3 (−7.7); editor without linting → 15.0 (−3.0); no search → 15.7 (−2.3); file viewer at 30 lines → 14.3 (−3.7) and full-file viewer → 12.7 (−5.3) vs the 100-line default; full history (vs last-5 observations) → 15.0 (−3.0). The custom ACI solves **10.7 percentage points** more than a baseline agent using only the default shell; an LM-friendly ACI gives a 64% relative improvement.
- **Findings — ReAct (Yao, Zhao, Yu, Du, Shafran, Narasimhan, Cao 2022/ICLR 2023, "ReAct: Synergizing Reasoning and Acting in Language Models")**:
  - **Mechanism**: prompt the LM to generate **reasoning traces and task-specific actions in an interleaved manner** (thought → action → observation, repeat). Reasoning traces help induce/track/update action plans and handle exceptions; actions let the model query external sources (knowledge base, environment) to gather info — closing the loop CoT-alone lacks.
  - **Results (locators)**: on **HotpotQA** (QA) and **Fever** (fact verification), ReAct + a simple Wikipedia API reduces hallucination/error-propagation vs CoT and yields more interpretable trajectories. On two decision-making benchmarks, **ALFWorld** and **WebShop**, ReAct beats imitation/RL methods by **+34%** and **+10%** absolute success rate respectively, with only 1–2 in-context examples.
- **Sources**:
  - J. Yang et al., "SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering," NeurIPS 2024 — (local: download/swe-agent-2024.pdf)
  - S. Yao et al., "ReAct: Synergizing Reasoning and Acting in Language Models," ICLR 2023 — (local: download/react-2022.pdf)
- **Confidence**: high
- **Gaps**: SWE-bench *Verified* (the OpenAI-curated 500-instance split, released Aug 2024 — after this SWE-agent paper) is not reported in this paper; SWE-agent's later Verified numbers would need a separate web/leaderboard source.

### Q5: SWE-agent vs other systems — AutoCodeRover, OpenHands/OpenDevin, Aider, and "scaffolding vs base-model capability"

- **AutoCodeRover (Zhang, Ruan, Fan, Roychoudhury 2024, "AutoCodeRover: Autonomous Program Improvement")**:
  - **Approach**: software-engineering-oriented; operates on a **program representation (abstract syntax tree)** rather than treating the project as a bag of files. Code-search APIs run locally over the **AST** to retrieve class signatures / method bodies by program structure (classes/methods), via **iterative search**. When a test-suite exists, **spectrum-based fault localization (SBFL)** further sharpens the retrieved context. Outputs a patch.
  - **Numbers (SWE-bench Lite, 300 issues)**: **19% efficacy** (resolved **57** GitHub issues), pass@1 **12.63%** avg and pass@3 **18.77%** — reported as higher than the then-current SWE-agent and Devin, at significantly lower cost and ~4 minutes/issue. Base model: **GPT-4 (gpt-4-0125-preview)**. Date: April 2024.
- **OpenHands / OpenDevin (Wang et al. 2024, "OpenHands: An Open Platform for AI Software Developers as Generalist Agents," ICLR 2025)**:
  - **What it is**: a **platform** (formerly OpenDevin, MIT license, community project — 2.1K contributions from 188+ contributors) for building agents that act like a human developer: write code, use a command line, browse the web. Architecture: an **event-stream** abstraction connecting interfaces/agents/environments; a **docker-sandboxed** runtime with bash shell, web browser, and IPython server; agent-skills library; multi-agent collaboration; built-in evaluation harness over 15 benchmarks (SWE-Bench, WebArena, HumanEvalFix, etc.). Ships a strong generalist **CodeActAgent** (CodeAct architecture, Wang et al. 2024a) with web-browsing and code-editing specialists (the latter citing SWE-agent / Yang et al. 2024).
  - **Numbers (SWE-bench Lite, default split)**: **CodeActAgent v1.8 with claude-3.5-sonnet → 26% resolve rate**, "competitive compared to other open-source SWE specialists." Date: July 2024 (ICLR 2025 camera-ready).
- **Aider (web/docs — no acquired paper)**:
  - **What it is**: AI pair-programming tool in the terminal, working inside a git repo with reliable LLM **code editing** (diff / diff-fenced / architect edit formats) and a **repository map** (static analysis of repo layout/structure to feed the LLM). User-directed rather than fully autonomous-agent.
  - **Numbers**: **SWE-bench Lite 26.3% (79/300), pass@1, no hints, GPT-4o + Claude 3 Opus (alternating attempts), 22 May 2024**; SWE-bench (main/full) **18.9%**, pass@1, 2 June 2024.
- **Scaffolding vs base-model capability (synthesis from primary sources)**: SWE-agent's central, directly-measured claim is that the *interface / scaffolding* materially changes resolved rate for a **fixed** base model — its ACI ablation moves Lite from 18.0 down to 10.3 just by swapping the custom editor for the raw shell (a 64% relative swing), and the custom ACI beats a default-shell agent by **10.7 pp**. AutoCodeRover makes the orthogonal point: a structure-aware (AST + SBFL) scaffold beats a more generic shell agent at lower cost on the same base GPT-4. The cross-system comparison is confounded by changing base models over time (GPT-4 Turbo → GPT-4o → Claude 3 Opus → Claude 3.5 Sonnet) — e.g. OpenHands' jump to 26% rides partly on claude-3.5-sonnet — so resolved-% deltas across these papers conflate scaffold and base-model improvements.
- **Sources**:
  - Y. Zhang, H. Ruan, Z. Fan, A. Roychoudhury, "AutoCodeRover: Autonomous Program Improvement," ISSTA 2024 — (local: download/autocoderover-2024.pdf)
  - X. Wang et al., "OpenHands: An Open Platform for AI Software Developers as Generalist Agents," ICLR 2025 — (local: download/openhands-2024.pdf)
  - SWE-agent — (local: download/swe-agent-2024.pdf) [for the scaffolding ablation]
  - Aider, "How aider scored SOTA 26.3% on SWE Bench Lite" (2024-05-22) and "Aider is SOTA for both SWE Bench and SWE Bench Lite" (2024-06-02); leaderboard docs — (web: https://aider.chat/2024/05/22/swe-bench-lite.html ; https://aider.chat/2024/06/02/main-swe-bench.html ; https://aider.chat/docs/leaderboards/)
- **Confidence**: high for AutoCodeRover/OpenHands/SWE-agent (local); medium for Aider (web blog/docs, self-reported, not peer-reviewed)
- **Gaps**: All SWE-agent-era numbers predate **SWE-bench Verified** (Aug 2024); current Verified leaderboard standings for these systems would need a fresh web/leaderboard source. Aider's numbers are self-reported on a self-run harness.

### Q2: Low-latency serving for autocomplete — latency budgets, caching, speculative decoding, quantization

Note: this topic is dominated by vendor engineering blogs; primary peer-reviewed sources are thin. The one *primary* anchor is the FIM paper's KV-cache point; the rest are flagged (web). Kept concise per scope.

- **Findings — primary anchor**:
  - KV-cache friendliness is an explicit serving-time design lever for code completion: the FIM paper's **SPM mode** reorders to suffix-prefix-middle precisely so that appending tokens to the prefix does not invalidate the suffix's cached keys/values during inference (Bavarian et al. 2022, §3.1). This is the in-survey-corpus primary citation for "caching considerations specific to FIM-based code assistants."
- **Findings — latency budgets (web, vendor)**:
  - A full editor→server→model→editor round trip for inline completion targets roughly **~300 ms**; GitHub reported a median completion latency drop from **1.4 s → 0.7 s** via speculative-decoding-class improvements. Network RTT becomes the dominant residual term once model latency is cut (e.g. ~100 ms extra SF↔Virginia). GitHub's new custom completions model reports **35% latency reduction, 3x higher tokens/s throughput, 12% higher acceptance, 20% more accepted/retained characters**, and is **FIM-trained** (prefix/suffix awareness).
- **Findings — speculative decoding (web)**:
  - Speculative decoding (a small draft model proposes tokens, the large target verifies them in one forward pass) is the headline latency lever, cited at roughly 2–3x. Cursor's "Fast Apply"/speculative-edits uses the *existing source code as draft tokens* to push a 70B model to ~1,000 tokens/s (~13x reported speedup) — the code-assistant-specific twist is that the edit target overlaps heavily with existing code, so the draft is nearly free.
- **Findings — caching & quantization (web)**:
  - Caching operates at two layers for code assistants: (a) **prompt/KV caching** of the long repo-context prefix (GitHub uses prompt caching across Anthropic/Bedrock/Vertex backends), and (b) **embedding/chunk caching** for retrieval — AST-aware chunks (100–250 tokens) cached by file hash in SQLite, with a debounced re-index invalidating only changed files. Quantization is the standard throughput/footprint lever but no code-assistant-specific quantization primary source was acquired.
- **Sources**:
  - Bavarian et al. 2022 (FIM, SPM KV-cache rationale) — (local: download/fim-bavarian-2022.pdf)
  - "The road to better completions: building a faster, smarter GitHub Copilot with a new custom model," GitHub Blog — (web: https://github.blog/ai-and-ml/github-copilot/the-road-to-better-completions-building-a-faster-smarter-github-copilot-with-a-new-custom-model/)
  - "How Cursor built Fast Apply using the Speculative Decoding API," Fireworks AI — (web: https://fireworks.ai/blog/cursor)
  - GitHub Docs, "Hosting of models for GitHub Copilot" / prompt-caching — (web: https://docs.github.com/en/copilot/reference/ai-models/model-hosting)
  - NVIDIA Technical Blog, "An Introduction to Speculative Decoding for Reducing Latency in AI Inference" — (web: https://developer.nvidia.com/blog/an-introduction-to-speculative-decoding-for-reducing-latency-in-ai-inference/)
- **Confidence**: medium (FIM/SPM caching point is high/primary; latency-budget and speculative-decoding figures are vendor-blog/web, self-reported and version-dependent)
- **Gaps**: No acquired *primary* source for (a) a quantified code-autocomplete latency budget, (b) quantization tradeoffs specific to code models, (c) speculative decoding as a peer-reviewed method (the original Leviathan et al. 2023 / Chen et al. 2023 speculative-decoding papers were not acquired — flag for §10 if a primary spec-decoding citation is needed; budget exhausted at 6/6 PDFs).
