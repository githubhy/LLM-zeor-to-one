### Q1: HumanEval (164) + MBPP (974) — construction, pass@k in practice, limitations

- **Findings**:
  - **HumanEval** (from Codex paper, §2.1–2.2): **164 hand-written** programming problems, created so functional correctness could be measured by unit tests rather than match-based metrics (BLEU). Each problem includes a **function signature, docstring, body, and several unit tests**, averaging **7.7 tests per problem**. Problems assess "language comprehension, algorithms, and simple mathematics," some comparable to simple software-interview questions. All hand-written specifically to avoid leakage from training data ("not programmatically copied from existing sources").
  - **pass@k metric** (Codex §2): k samples generated per problem; problem counted solved if *any* sample passes the unit tests; report total fraction solved. Naive estimator (generate k, fraction solved) is high-variance. The paper's **unbiased estimator** generates n ≥ k samples, counts c ≤ n correct, and computes pass@k := E_problems[ 1 − C(n−c, k)/C(n, k) ]. In this paper they use **n = 200 samples, k ≤ 100**. The paper notes a numerically stable evaluation of this product is needed (direct computation overflows). Metric originally from Kulal et al. (2019).
  - **MBPP** (Austin et al., "Program Synthesis with Large Language Models", §2.1): "Mostly Basic Programming Problems," **974 tasks**, short Python programs from NL descriptions, "solvable by entry-level programmers." Construction: **crowd-sourced** — participants wrote a short problem statement, a single self-contained Python function, and **three test cases** (semantic-correctness checks) plus a ground-truth solution passing all three. Models evaluated 244M–137B params in **few-shot and fine-tuning** regimes.
  - **MBPP sanitized/edited subset**: authors "manually inspected, edited, and pruned a subset," yielding **426 hand-verified questions** ("edited dataset"), with standard signatures and disambiguated specs.
  - **MBPP headline (limitation context)**: largest model (no code fine-tune) solves **59.6%** of MBPP via few-shot with a well-designed prompt; fine-tuning adds ~**10 percentage points** across most sizes. Performance scales **log-linearly** with model size. On the 100 problems in both original and edited sets, few-shot performance rises from **63%** (original) — evidence the original crowd-sourced specs are noisier/ambiguous (a construction limitation).
- **Sources**:
  - M. Chen et al., "Evaluating Large Language Models Trained on Code" (Codex), 2021 — (local: download/chen-codex-evaluating-llms-code-2021.pdf)
  - J. Austin et al., "Program Synthesis with Large Language Models" (MBPP), 2021 — (local: download/mbpp-austin-2021.pdf)
- **Confidence**: high
- **Gaps**: HumanEval's "limited test coverage" critique is addressed quantitatively in Q2 (EvalPlus), not in the original Codex paper.

### Q2: Test-adequacy critique + harder functional benchmarks (EvalPlus/HumanEval+, BigCodeBench, ClassEval)

- **Findings**:
  - **EvalPlus / HumanEval+** (Liu et al., "Is Your Code Generated by ChatGPT Really Correct?", NeurIPS 2023):
    - **Motivating critique**: existing benchmarks have **on average < 10 tests per problem**, and those tests are "too simple to fully explore the functionality of the code or corner cases" → inflated pass rates from undetected wrong code.
    - **Test augmentation**: EvalPlus uses an automated test-input generator combining **LLM-based + mutation-based** strategies (seeds from ChatGPT, then type-aware mutation). It extends the HumanEval test cases by **80×** to build **HumanEval+** (still the same **164 problems**).
    - **pass@k drop (exact numbers)**: across all 26 evaluated LLMs, sizes, and k, almost all pass@k results **consistently drop** on HumanEval+ vs base HumanEval. Headline reductions: **up to 23.1% (pass@1⋆ greedy) / 19.3% (pass@1) / 24.9% (pass@10) / 28.9% (pass@100)**. Abstract summarizes as "reducing pass@k by up-to 19.3–28.9%."
    - **Mis-ranking from test insufficiency**: with the stronger tests, the leaderboard reorders — e.g. WizardCoder-CodeLlama and Phind-CodeLlama now outperform ChatGPT, a flip that the weak base tests had hidden. (Strong argument that test adequacy, not just problem difficulty, drives reported scores.)
    - Also ships a **reduced/minimal test-suite** (HumanEval+-Min via greedy set cover, Table 4) preserving most of the bug-catching power at far fewer tests.
  - **BigCodeBench** (Zhuo et al., ICLR 2025): **1,140 fine-grained tasks** requiring LLMs to invoke **multiple function calls as tools** from **139 libraries across 7 domains**. Each task has **5.6 test cases** on average with **99% average branch coverage** (rigorous execution-based eval). Two variants: **BigCodeBench-Complete** (docstring completion) and **BigCodeBench-Instruct** (NL-instruction variant; docstrings auto-transformed to short instructions). Evaluated **60 LLMs**.
    - **Real-world gap headline**: best model GPT-4o solves **merely 60% on BigCodeBench-Complete and < 50% on BigCodeBench-Instruct**; scores **up to 60% vs human performance of 97%** — a ~37-point gap showing LLMs cannot follow complex instructions to use function calls precisely. Uses **calibrated Pass@1** (adds missing setup so import/setup omissions aren't counted as failures); Complete↔Instruct rank correlation Pearson r = 0.982.
  - **ClassEval** (Du et al., arxiv:2308.01861): **100 class-level Python code-generation tasks** (~500 person-hours hand-built), testing **class-level** generation (inter-method dependencies, class structure) vs standalone method-level (HumanEval/MBPP). Three generation strategies: **holistic** (whole class at once; best for GPT-4/GPT-3.5), **incremental**, and **compositional** (method-by-method; better for weaker instruction-followers). Headline: **all LLMs perform much worse on class-level than on method-level**; method-level ability "cannot equivalently reflect class-level coding ability." (Exact pass@1 numbers not in abstract — full text not acquired.)
- **Sources**:
  - J. Liu, C. S. Xia, Y. Wang, L. Zhang, "Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of LLMs for Code Generation" (EvalPlus / HumanEval+), NeurIPS 2023 — (local: download/evalplus-2023.pdf)
  - T. Y. Zhuo et al., "BigCodeBench: Benchmarking Code Generation with Diverse Function Calls and Complex Instructions," ICLR 2025 — (local: download/bigcodebench-2024.pdf)
  - X. Du et al., "ClassEval: A Manually-Crafted Benchmark for Evaluating LLMs on Class-level Code Generation," 2023 — (abstract-only: arxiv:2308.01861)
- **Confidence**: high (EvalPlus, BigCodeBench); medium (ClassEval — abstract-only, no exact pass@1)
- **Gaps**: ClassEval exact pass@1 numbers and class/method counts beyond "100 classes" not captured (full text not in budget). MBPP+ (EvalPlus also augments MBPP by ~35×) numbers not extracted — only HumanEval+ headline pulled.

### Q3: SWE-bench family (SWE-bench, Verified, Lite) — construction, "resolved", vs HumanEval

- **Findings**:
  - **SWE-bench** (Jimenez, Yang et al., "Can Language Models Resolve Real-World GitHub Issues?", ICLR 2024): **2,294 software-engineering task instances** drawn from **real GitHub issues + their merged pull requests** across **12 popular Python repositories** (each >90% Python). Task: given a codebase snapshot + issue description, the model edits the repo to produce a **patch**, evaluated against **real unit tests**.
  - **3-stage construction pipeline** (Fig. 2):
    1. **Scrape** PRs from the 12 repos.
    2. **Attribute filter**: keep PRs that (a) resolve an issue and (b) contribute tests.
    3. **Execution-based filter**: apply the PR's test content, log test results before/after the PR's non-test content; **discard any instance lacking ≥1 test whose status flips fail→pass** ("fail-to-pass test"); also discard installation/runtime-error instances.
  - **What "resolved" means**: an instance is resolved when, after applying the model's patch, the **fail-to-pass tests pass** AND the previously passing tests still pass (execution-based, not text-match). Gold-patch stats: avg **9.1 fail-to-pass tests**, **120.8 total tests** per instance; gold patch edits avg **32.8 lines, 1.7 files, 3 functions** — so resolving frequently requires coordinating changes across multiple functions/classes/files in a large codebase (mean **3,010 non-test files / 438K lines**; issue text mean 195 words).
  - **Headline difficulty**: best model in the paper, **Claude 2, resolves a mere 1.96%** of issues; proprietary models + fine-tuned SWE-Llama solve only the simplest issues. (Establishes the huge gap vs saturated function-level benchmarks.)
  - **SWE-bench Lite**: original paper creates a **Lite subset of 300 instances** selected to be more self-contained / cheaper to run while preserving the task format.
  - **SWE-bench Verified** (OpenAI, 2024, web): a **500-instance human-validated subset**, annotated by **professional software developers**, to remove flawed instances. Motivation/evidence from annotation: **38.3% of original samples flagged for underspecified problem statements**; **61.1% flagged for unit tests that may unfairly mark valid solutions wrong**; overall **68.3% of SWE-bench samples were filtered out** (underspecification / unfair tests / other). Effect: scores roughly **double** — **GPT-4o resolves 33.2% on Verified** vs ~16% on original — i.e. much of the original "difficulty" was benchmark noise, not model inability.
  - **vs HumanEval**: SWE-bench measures **repository-level, agentic issue resolution** (multi-file patch on a real large codebase, real regression tests, realistic issue text) rather than HumanEval's **single self-contained function from a docstring**. It is far harder, contamination-resistant by realism, and continuously updatable with new GitHub instances "with minimal human intervention."
- **Sources**:
  - C. E. Jimenez, J. Yang et al., "SWE-bench: Can Language Models Resolve Real-World GitHub Issues?", ICLR 2024 — (local: download/swe-bench-2023.pdf)
  - OpenAI, "Introducing SWE-bench Verified," 2024 — (web: https://openai.com/index/introducing-swe-bench-verified/)
- **Confidence**: high (SWE-bench, Lite — from local PDF); high (Verified — OpenAI announcement, corroborated by HF dataset card / multiple sources)
- **Gaps**: Verified numbers are from the OpenAI blog (403 on direct fetch; relayed via search snippet of the same page + HF card). The exact original-SWE-bench GPT-4o number ("~16%") is the doubling implication, not pulled verbatim — treat "doubles" as the cited claim.

### Q4: Contamination & saturation (LiveCodeBench, CodeContests/AlphaCode)

- **Findings**:
  - **LiveCodeBench** (Jain, Han et al., "Holistic and Contamination Free Evaluation of LLMs for Code," 2024):
    - **Method (contamination-resistance via time-windowing)**: continuously collects *new* problems over time from **3 competition platforms — LeetCode, AtCoder, CodeForces** — and tags every problem with its **release date**. Because each problem has a release date, evaluation can be restricted to **time windows after a model's training cutoff**, guaranteeing the model never saw the exact problem. (Notes that contamination is hard to detect by string match since it "can be evaded by rephrasing.")
    - **Size**: hosts **511 problems** published **May 2023 → May 2024** at paper time ("over five hundred"); difficulty tagged Easy/Medium/Hard via contest rating brackets; AtCoder public+hidden tests reused directly.
    - **Holistic scope**: not just code generation — **four scenarios**: code generation, **self-repair**, **code execution**, and **test output prediction**. Evaluated **34 instruction-tuned LLMs**.
    - **Evidence of contamination in older setups (the headline)**: model performance **starkly drops on problems released after the model's cutoff**. Concretely, **DeepSeek-Instruct** performs much worse on LeetCode problems released **after Aug/Sep 2023** (its release window), and **GPT-4-O drops on LeetCode problems released since Nov 2023** (its stated cutoff). The before/after-cutoff gap is the direct signal that high scores on older static benchmarks partly reflect memorization, not capability ("potential overfitting in existing benchmarks").
  - **AlphaCode / CodeContests** (Li et al., "Competition-Level Code Generation with AlphaCode," DeepMind 2022):
    - **Competitive-programming eval**: AlphaCode achieved an average ranking of **top 54.3%** in simulated Codeforces competitions with **>5,000 participants** (estimated Codeforces rating **1238**, > 72% of users who competed in the last 6 months). Solve metric = **n@k** generalization of pass@k: sample k, *filter* to n ≤ k submissions (filtering only uses info a real competitor has — e.g. the example tests), solved if any of the n passes all hidden tests. Used **≤10 submissions per problem** (human-comparable).
    - **False-positive problem (the saturation critique)**: competitive-programming problems with sparse public tests have **high false-positive rates — 30% or more (paper says 30–60% in existing datasets) of programs pass all available tests but are not actually correct**, making such metrics "unreliable for measuring research progress." Also **"slow positives"**: correct-but-inefficient solutions of the wrong complexity class that pass because tests don't enforce time/memory limits.
    - **CodeContests filtering / mitigation**: the CodeContests fine-tuning+eval dataset (a) **splits temporally** so all training data predates all eval problems (anti-contamination), and (b) **adds mutation-generated extra tests** (bit-flips on binary inputs, increment/decrement integers, swap/change characters in strings) verified by running **30 correct solutions** on each mutated input. This **reduces the false-positive rate from 30–60% down to just 4%**. Dataset splits: Train 13,328 / Valid 117 / Test 165 problems; mean generated tests ~190+ on valid/test vs ~9–13 hidden tests. Best model solves **34.2%** of held-out problems at ≤10 submissions (vs ~1–5% previously reported on noisier setups).
- **Sources**:
  - N. Jain, K. Han et al., "LiveCodeBench: Holistic and Contamination Free Evaluation of LLMs for Code," 2024 — (local: download/livecodebench-2024.pdf)
  - Y. Li et al., "Competition-Level Code Generation with AlphaCode," Science / DeepMind 2022 — (local: download/alphacode-2022.pdf)
- **Confidence**: high (both local full text)
- **Gaps**: LiveCodeBench problem count grows post-publication (511 is the May-2024 snapshot); cite as "over 500 at release." AlphaCode "30% or more" vs "30–60%" — both phrasings appear in the paper; the 4% post-filter figure is unambiguous.

### Q5: Methodology — execution harness design, LLM-as-judge for code, benchmark-vs-real-world gap

- **Findings**:
  - **Execution-based eval is the standard for code** (Codex §2.1): purely match-based metrics (exact match, BLEU) are inadequate because "functionally equivalent" programs get low BLEU and functionally wrong programs can score high — code is graded by **running it against unit tests** (functional correctness), the paradigm pass@k formalizes. Every benchmark in this cluster (HumanEval, MBPP, EvalPlus, BigCodeBench 99% branch coverage, SWE-bench, LiveCodeBench, CodeContests) is execution-based.
  - **Sandboxing untrusted code** (Codex §2.3): model-generated code is untrusted and may be adversarial/destructive, so it must run in a **sandbox**. Codex used the **gVisor container runtime** (kernel-level syscall interception) as the main host-protection layer because ordinary Docker shares host resources and a malicious container could compromise the host; goals: prevent the program from modifying/persisting on the host, accessing sensitive resources, or exfiltrating data over the network.
  - **Repo-level harness design** (SWE-bench §A.4, evaluation procedure): pipeline is **(1) model generates a patch → (2) Apply patch to the repo snapshot → (3) Run the task's test script → (4) Log results** and compare fail-to-pass / pass-to-pass status. Each task is paired with the exact package versions / environment needed to **reproduce** the issue (per-instance reproducible environments are what make execution-based repo-level eval feasible — later SWE-bench tooling ships per-instance Docker images, though the original paper describes the apply→run→log harness abstractly).
  - **LLM-as-judge for code** (web synthesis, not a single local source): used as a **complement** to execution-based eval to capture qualities unit tests miss — **readability, style conformance, and subtle semantic issues that don't trip a test**. Reliability is **task-dependent and currently weak for correctness**: state-of-the-art models "cannot reliably determine if a piece of code works without executing it," and the paradigm carries documented **bias / misalignment / fairness** concerns. Emerging hybrids have the LLM *simulate* line-by-line execution of synthesized test cases to approach unit-test-level reliability. Consensus: LLM-as-judge augments, does not replace, execution-based testing for correctness gates.
  - **Documented benchmark-vs-real-world gap** (cross-cutting, drawn from the local sources):
    - **Function-level saturation vs SE reality**: HumanEval/MBPP are short, self-contained functions and are now largely saturated, yet on **real GitHub issues (SWE-bench) the best model in the original paper resolved only 1.96%** — direct evidence that high HumanEval scores do not transfer to repository-scale engineering.
    - **Instruction-following / library-usage gap**: BigCodeBench shows the best model (GPT-4o) at **~60% vs 97% human**, i.e. LLMs cannot reliably compose real library/function-call workflows even when they ace toy functions.
    - **Test-adequacy inflation**: EvalPlus's **80× test augmentation drops pass@k by up to ~19–29%** and reorders leaderboards — reported scores were partly artifacts of weak test suites, not real correctness.
    - **Contamination inflation**: LiveCodeBench's before/after-cutoff drops (DeepSeek, GPT-4-O) show that part of static-benchmark performance is memorization, overstating real generalization.
    - **Benchmark noise**: SWE-bench Verified found **68.3% of original SWE-bench instances were flawed** (underspecified / unfair tests), so even the "hard real-world" numbers were noisy — scores roughly **doubled** once cleaned. The gap cuts both ways: benchmarks can both *over*state (contamination, weak tests) and *under*state (broken instances) real ability.
- **Sources**:
  - M. Chen et al., "Evaluating Large Language Models Trained on Code" (sandbox/gVisor, functional correctness), 2021 — (local: download/chen-codex-evaluating-llms-code-2021.pdf)
  - C. E. Jimenez, J. Yang et al., "SWE-bench" (apply→run→log harness, reproducible envs), ICLR 2024 — (local: download/swe-bench-2023.pdf)
  - J. Liu et al., "EvalPlus / HumanEval+" (test-adequacy inflation), NeurIPS 2023 — (local: download/evalplus-2023.pdf)
  - T. Y. Zhuo et al., "BigCodeBench" (instruction/library gap, 99% branch coverage), ICLR 2025 — (local: download/bigcodebench-2024.pdf)
  - N. Jain et al., "LiveCodeBench" (contamination inflation), 2024 — (local: download/livecodebench-2024.pdf)
  - OpenAI, "Introducing SWE-bench Verified" (benchmark-noise 68.3% filtered), 2024 — (web: https://openai.com/index/introducing-swe-bench-verified/)
  - LLM-as-judge-for-code reliability — (web: synthesis of arxiv:2502.12468 "MCTS-Judge: Test-Time Scaling in LLM-as-a-Judge for Code Correctness", plus the code-intelligence survey arxiv:2511.18538; task-dependent / not-yet-reliable claim is the load-bearing point)
