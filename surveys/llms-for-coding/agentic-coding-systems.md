## 12 Agentic Coding Systems

<a id="p-12-agentic-coding-systems-1"></a><!-- para:12-agentic-coding-systems-1 --> Agents are where the field's center of gravity moved (Section 3). An agent wraps a code model in a loop: it observes a repository, takes an action (read a file, edit code, run tests, search), reads the result, and iterates until a task is done. The unit of work is no longer "a function from a docstring" but "a task in a real codebase," which is exactly what SWE-bench measures (Section 13). This section covers the loop, the systems that implement it, and the central methodological question — how much of the performance is the model versus the scaffolding.

<!-- sec:12.1 -->
### <a id="sec-12.1"></a>12.1 The Agent Loop: ReAct and the Agent-Computer Interface

<a id="p-121-the-agent-loop-react-and-the-agent-computer-interface-1"></a><!-- para:121-the-agent-loop-react-and-the-agent-computer-interface-1 --> The reasoning substrate of most coding agents is ReAct: prompt the model to interleave reasoning traces with task actions, so that thoughts plan and track the work while actions query the environment and gather observations <!-- cite:31 --> [[31]](references.md#ref-31). ReAct's original results — beating imitation and RL baselines by 34% and 10% absolute on the ALFWorld and WebShop decision-making benchmarks with only a couple of in-context examples — established that reasoning plus acting beats either alone <!-- cite:31 --> [[31]](references.md#ref-31).

<a id="p-121-the-agent-loop-react-and-the-agent-computer-interface-2"></a><!-- para:121-the-agent-loop-react-and-the-agent-computer-interface-2 --> SWE-agent makes the sharper claim that the *interface* matters as much as the model. Its thesis is that an LM agent is a new kind of end user, and a purpose-built **agent-computer interface (ACI)** — the commands the agent can issue and the format of the feedback it receives — substantially outperforms handing the agent a raw shell <!-- cite:32 --> [[32]](references.md#ref-32). Its four design principles: actions should be simple and few; actions should be compact and efficient so one step makes real progress; feedback should be informative but concise; and guardrails (such as a syntax checker wired into the edit command) should catch and shorten error loops <!-- cite:32 --> [[32]](references.md#ref-32). The payoff is measured directly: with GPT-4 Turbo, SWE-agent resolves 12.5% of the full SWE-bench test set and 18.0% of SWE-bench Lite, versus 3.8% for the prior non-interactive retrieval system, and it ports across models (10.5% with Claude 3 Opus) <!-- cite:32 --> [[32]](references.md#ref-32). Its ablations isolate the interface: replacing the custom editor with the raw shell drops Lite from 18.0% to 10.3%, and the custom ACI solves 10.7 percentage points more than a default-shell agent — a 64% relative improvement attributable to the interface, not the model <!-- cite:32 --> [[32]](references.md#ref-32).

### <a id="sec-12.2"></a>12.2 A Spectrum of Systems

<a id="p-122-a-spectrum-of-systems-1"></a><!-- para:122-a-spectrum-of-systems-1 --> Coding agents span a design spectrum from structure-aware to generalist to human-in-the-loop:

- <a id="p-122-a-spectrum-of-systems-2"></a><!-- para:122-a-spectrum-of-systems-2 --> **AutoCodeRover** operates on a program's abstract syntax tree rather than treating it as a bag of files, using AST-aware code-search APIs and, when a test suite exists, spectrum-based fault localization to focus context. On SWE-bench Lite it resolves 19% (57 issues) at roughly four minutes per issue using GPT-4, at lower cost than a generic shell agent <!-- cite:33 --> [[33]](references.md#ref-33).
- **OpenHands** (formerly OpenDevin) is an open *platform* rather than a single agent: an event-stream architecture connects agents, interfaces, and a Docker-sandboxed runtime with a shell, browser, and Python server, plus a built-in evaluation harness. Its generalist CodeActAgent reports a 26% SWE-bench Lite resolve rate with Claude 3.5 Sonnet <!-- cite:34 --> [[34]](references.md#ref-34).
- **Aider** is a terminal pair-programming tool emphasizing reliable code editing (diff-style edit formats) and a static "repository map" to feed structure to the model; it reports 26.3% on SWE-bench Lite, though as self-reported results on a self-run harness rather than peer-reviewed numbers <!-- cite:48 --> [[48]](references.md#ref-48).

### <a id="sec-12.3"></a>12.3 Scaffolding versus Base Model

<a id="p-123-scaffolding-versus-base-model-1"></a><!-- para:123-scaffolding-versus-base-model-1 --> The central caveat in reading any of these numbers is that resolved-rate comparisons across systems and time *conflate two improvements*: a better scaffold and a better base model. SWE-agent's ablations show that, holding the base model fixed, the scaffold alone moves the resolved rate by more than ten points <!-- cite:32 --> [[32]](references.md#ref-32); AutoCodeRover shows a structure-aware scaffold beating a generic one on the same model <!-- cite:33 --> [[33]](references.md#ref-33). But cross-system comparisons are confounded because the base model changes underneath — GPT-4 Turbo, then GPT-4o, then Claude 3 Opus, then Claude 3.5 Sonnet — so OpenHands's 26% versus SWE-agent's 18% is partly a stronger model, not purely a better harness. The practical implication, revisited in Section 15, is that headline SWE-bench numbers should be read as *(model × scaffold)* products with a date attached, never as a property of either component alone. The frontier trend is that as base models acquire stronger reasoning (Section 9), the scaffold can become thinner — the same lesson the o-series drew from competitive programming <!-- cite:26 --> [[26]](references.md#ref-26).
