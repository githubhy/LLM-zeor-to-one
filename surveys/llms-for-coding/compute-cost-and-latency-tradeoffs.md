## 14 Compute, Cost, and Latency Tradeoffs

<a id="p-14-compute-cost-and-latency-tradeoffs-1"></a><!-- para:14-compute-cost-and-latency-tradeoffs-1 --> Three distinct resources govern a code model's economics: the training compute spent once, the inference cost paid per use, and the latency a user experiences. Which one binds depends entirely on the deployment target (Section 5). This section makes the tradeoffs concrete and explains why the "best" code model is an ill-posed notion without naming the axis.

<!-- sec:14.1 -->
### <a id="sec-14.1"></a>14.1 Model Size versus Capability

<a id="p-141-model-size-versus-capability-1"></a><!-- para:141-model-size-versus-capability-1 --> Raw parameter count is a poor proxy for code capability, and two mechanisms decouple them. The first is **mixture-of-experts sparsity**: DeepSeek-Coder-V2 has 236B total parameters but activates only 21B per token, with a "Lite" variant of 16B total and 2.4B active; the Lite model's 2.4B active parameters match the dense DeepSeek-Coder-33B on Python completion, an order-of-magnitude reduction in active compute at comparable capability <!-- cite:43 --> [[43]](references.md#ref-43). The second is **deliberate over-training** (Section 7): StarCoder 2 trains small models far past the compute-optimal token count specifically to make *inference* cheap, accepting higher training cost for a smaller, faster deployed model <!-- cite:9 --> [[9]](references.md#ref-9). Together these explain how open-weight models reached frontier-comparable code performance — DeepSeek-Coder-V2 is explicitly framed as closing the gap to closed models <!-- cite:43 --> [[43]](references.md#ref-43) — while remaining cheap enough to serve. The takeaway: for code, activation cost and data quality (Section 6) matter more than nominal size.

<!-- sec:14.2 -->
### <a id="sec-14.2"></a>14.2 The Cost of Test-Time Compute

<a id="p-142-the-cost-of-test-time-compute-1"></a><!-- para:142-the-cost-of-test-time-compute-1 --> Section 9 showed that correctness can be bought with inference compute — more samples, longer reasoning, self-repair loops. That purchase has a price, and the frontier increasingly reports accuracy *against a cost budget* rather than in isolation. The lever is visible in the pass@k gap: CodeRL rises from ~2% pass@1 to ~20% pass@1000 on APPS purely by sampling more <!-- cite:19 --> [[19]](references.md#ref-19), and AlphaCode's competition results require sampling thousands of candidates before filtering <!-- cite:40 --> [[40]](references.md#ref-40) — each additional sample is linear inference cost. Reasoning models (Section 9) spend hundreds to thousands of tokens of chain-of-thought per problem. Concretely, dated cost-versus-accuracy leaderboards for agentic SWE-bench tasks show a steep frontier: a high-end configuration resolving on the order of 46% of tasks at roughly USD 3 per task, versus a cheap model resolving about 24% at under USD 0.10 per task — a two-orders-of-magnitude cost spread for roughly double the solve rate (dated web estimates, with the verification caveats of Section 15) <!-- cite:49 --> [[49]](references.md#ref-49). The structural implication is that near-frontier agentic solve rates are bought with super-linear inference cost, so cost-per-task, not accuracy alone, is the figure that matters in production.

<!-- sec:14.3 -->
### <a id="sec-14.3"></a>14.3 Latency versus Throughput

<a id="p-143-latency-versus-throughput-1"></a><!-- para:143-latency-versus-throughput-1 --> The final axis is wall-clock, and it splits cleanly by deployment. **Inline autocomplete** is latency-bound: a completion must appear in well under a second, which favors small models, fill-in-the-middle, multi-query attention (StarCoder's explicit choice for fast batched inference <!-- cite:8 --> [[8]](references.md#ref-8)), caching, and speculative decoding (Section 10), with vendors reporting median completion latency cut from 1.4 to 0.7 seconds <!-- cite:27 --> [[27]](references.md#ref-27), <!-- cite:50 --> [[50]](references.md#ref-50). **Agents** are throughput- and total-cost-bound: a single task issues many calls over large contexts, so the relevant quantities are tokens-per-task and context-window cost, not first-token time, and a task that takes minutes and dollars is acceptable if it resolves a real issue. The same base model therefore lands at opposite ends of the optimization space depending on whether it autocompletes or acts. This is the quantitative backbone of the design guidance in Section 17: pick the model and serving stack to fit the binding resource, because no single configuration is simultaneously the cheapest, the lowest-latency, and the most capable.
