## 13 Evaluation and Benchmarks

<a id="p-13-evaluation-and-benchmarks-1"></a><!-- para:13-evaluation-and-benchmarks-1 --> How we measure code ability has shaped what the field optimizes, and the benchmark lineage is a story of each generation patching the previous one's blind spot. This section traces that lineage — from function-level pass@k to repository-scale issue resolution — and then steps back to the methodology and the persistent gap between a benchmark score and real-world competence. The governing principle, stated once: a leaderboard number is not a capability claim.

<!-- sec:13.1 -->
### <a id="sec-13.1"></a>13.1 The Foundational Function-Level Benchmarks

<a id="p-131-the-foundational-function-level-benchmarks-1"></a><!-- para:131-the-foundational-function-level-benchmarks-1 --> HumanEval (Section 3) is the origin point: 164 hand-written problems, each a function signature, docstring, body, and an average of 7.7 unit tests, written by hand specifically so functional correctness could be measured rather than text overlap <!-- cite:1 --> [[1]](references.md#ref-1). MBPP complements it with 974 crowd-sourced "mostly basic" Python problems, each with three test cases, plus a 426-problem hand-verified subset; its authors found few-shot performance on the cleaned subset rises noticeably, evidence that crowd-sourced specifications are themselves noisy <!-- cite:35 --> [[35]](references.md#ref-35). Both are scored by pass@k (the unbiased estimator of Section 3). Their virtue is simplicity and reproducibility; their vice is that they are short, self-contained, now largely saturated, and — as the next subsection shows — under-tested.

<!-- sec:13.2 -->
### <a id="sec-13.2"></a>13.2 The Test-Adequacy Critique

<a id="p-132-the-test-adequacy-critique-1"></a><!-- para:132-the-test-adequacy-critique-1 --> If a benchmark's tests are weak, wrong code passes and scores inflate. EvalPlus made this quantitative: existing benchmarks average fewer than ten tests per problem, so EvalPlus augments HumanEval's tests roughly 80-fold (via LLM-seeded, type-aware mutation) to build HumanEval+, keeping the same 164 problems but vastly stronger tests <!-- cite:36 --> [[36]](references.md#ref-36). Across 26 models, pass@k drops consistently — by up to 19.3% at pass@1 and up to 28.9% at pass@100 — and, crucially, the leaderboard *reorders*: models that the weak tests ranked below ChatGPT overtake it once the tests are adequate <!-- cite:36 --> [[36]](references.md#ref-36). Reported scores were partly artifacts of test insufficiency, not pure capability. BigCodeBench raises the bar on realism: 1,140 tasks requiring multiple function calls across 139 libraries in 7 domains, with about 5.6 tests per task at 99% average branch coverage; the best model (GPT-4o) reaches only about 60% versus 97% human performance, a roughly 37-point gap showing that models which ace toy functions cannot reliably compose real library workflows <!-- cite:37 --> [[37]](references.md#ref-37). ClassEval extends the difficulty axis to class-level generation (inter-method dependencies, class structure) and finds all models perform markedly worse than at method level, so method-level ability does not transfer to class-level coding <!-- cite:47 --> [[47]](references.md#ref-47).

<!-- sec:13.3 -->
### <a id="sec-13.3"></a>13.3 Repository-Level and Real-World: SWE-bench

<a id="p-133-repository-level-and-real-world-swe-bench-1"></a><!-- para:133-repository-level-and-real-world-swe-bench-1 --> SWE-bench reframed evaluation entirely: 2,294 task instances built from real GitHub issues and their merged pull requests across 12 popular Python repositories <!-- cite:38 --> [[38]](references.md#ref-38). Given a codebase snapshot and an issue, the model must produce a patch, evaluated by *execution* against the repository's real tests. Its three-stage construction scrapes PRs, keeps those that resolve an issue and contribute tests, and discards any instance lacking a test that flips from fail to pass after the gold patch <!-- cite:38 --> [[38]](references.md#ref-38). "Resolved" means the model's patch makes the fail-to-pass tests pass while not breaking previously passing tests — a behavioral criterion on a large codebase (gold patches average 32.8 edited lines across 1.7 files, against a mean of 3,010 non-test files) <!-- cite:38 --> [[38]](references.md#ref-38). On debut the best model (Claude 2) resolved a mere 1.96%, establishing the chasm between saturated function benchmarks and real engineering <!-- cite:38 --> [[38]](references.md#ref-38). Two derived sets followed: SWE-bench **Lite** (300 self-contained instances) and SWE-bench **Verified**, a 500-instance subset validated by professional developers after annotation found 38.3% of original instances underspecified and 61.1% with tests that could unfairly fail valid solutions — 68.3% filtered overall, after which scores roughly *doubled* (GPT-4o to 33.2%), showing much of the original "difficulty" was benchmark noise <!-- cite:46 --> [[46]](references.md#ref-46).

<!-- sec:13.4 -->
### <a id="sec-13.4"></a>13.4 Contamination and Saturation

<a id="p-134-contamination-and-saturation-1"></a><!-- para:134-contamination-and-saturation-1 --> Because benchmarks are public, models may have memorized them. LiveCodeBench addresses this by continuously collecting fresh problems from LeetCode, AtCoder, and Codeforces, each tagged with a release date, so evaluation can be restricted to problems published *after* a model's training cutoff <!-- cite:39 --> [[39]](references.md#ref-39). The diagnostic result: model performance drops sharply on post-cutoff problems — DeepSeek-Coder and GPT-4o both decline on problems released after their respective cutoffs — direct evidence that part of any static-benchmark score is memorization, not capability <!-- cite:39 --> [[39]](references.md#ref-39). AlphaCode's CodeContests illustrates a second failure mode: competitive-programming problems with sparse public tests have false-positive rates of 30–60% (programs that pass available tests but are wrong), which CodeContests reduces to about 4% by adding mutation-generated tests verified against known-correct solutions <!-- cite:40 --> [[40]](references.md#ref-40). AlphaCode itself reached an average ranking in the top 54.3% of Codeforces participants — impressive, but reported with the explicit caveat that naive metrics are "unreliable for measuring research progress" <!-- cite:40 --> [[40]](references.md#ref-40).

<!-- sec:13.5 -->
### <a id="sec-13.5"></a>13.5 Methodology and the Real-World Gap

<a id="p-135-methodology-and-the-real-world-gap-1"></a><!-- para:135-methodology-and-the-real-world-gap-1 --> Execution is the spine of all of this. Generated code is untrusted, so it runs in a sandbox — Codex used a gVisor container to prevent host compromise or data exfiltration <!-- cite:1 --> [[1]](references.md#ref-1) — and repository-level harnesses follow an apply-patch → run-tests → log-results procedure with per-instance reproducible environments <!-- cite:38 --> [[38]](references.md#ref-38). LLM-as-judge is used as a *complement* to execution for qualities tests miss (readability, style), but the literature is consistent that it is task-dependent and not reliable for *correctness* without execution; surveys treat it as an augmentation, not a replacement <!-- cite:44 --> [[44]](references.md#ref-44). Stepping back, the benchmark-to-reality gap cuts both ways. Benchmarks *overstate* ability through contamination (LiveCodeBench) and weak tests (EvalPlus), and they *understate* it through broken instances (the 68.3% SWE-bench filtering). The synthesis for a practitioner (Section 17): trust execution-based, contamination-resistant, adequately-tested benchmarks; read any single number with its date and its test-suite quality in mind; and treat the function-level classics as saturated sanity checks rather than measures of engineering competence.
