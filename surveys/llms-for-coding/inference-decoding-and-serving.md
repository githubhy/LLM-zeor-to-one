## 10 Inference, Decoding, and Serving

<a id="p-10-inference-decoding-and-serving-1"></a><!-- para:10-inference-decoding-and-serving-1 --> A trained model is not yet a product. Serving it for code adds constraints general chat does not: completions must be syntactically valid, they must arrive within a latency budget that autocomplete makes brutal, and the surrounding repository context can be enormous. This section covers the three levers — constrained decoding, latency engineering, and the serving-shape choice — that turn a checkpoint into an assistant.

<!-- sec:10.1 -->
### <a id="sec-10.1"></a>10.1 FIM Inference and Constrained Decoding

<a id="p-101-fim-inference-and-constrained-decoding-1"></a><!-- para:101-fim-inference-and-constrained-decoding-1 --> A model trained with fill-in-the-middle is *prompted* at completion time by encoding the prefix and suffix in the trained layout and letting the model generate the middle. In the prefix-suffix-middle scheme the model is fed everything up to the middle sentinel and samples until it emits an end-of-text token signaling it has joined prefix to suffix; failure to emit that token signals an unsuccessful join <!-- cite:5 --> [[5]](references.md#ref-5). This is the inference-time counterpart of the training transform of Section 4.

<a id="p-101-fim-inference-and-constrained-decoding-2"></a><!-- para:101-fim-inference-and-constrained-decoding-2 --> Beyond prompting, code benefits from *constraining* the decoder to produce only valid output. PICARD demonstrates the idea for SQL: during beam search it incrementally parses each candidate's detokenized output and rejects tokens that would make the query inadmissible, with checking modes ranging from lexing to full parsing with schema-aware guards <!-- cite:28 --> [[28]](references.md#ref-28). The effect is concrete — without it about 12% of a strong model's generated SQL queries fail with an execution error; with it only about 2% are unusable — and constraining also raises accuracy, lifting the base model to 75.5% exact-set-match and 79.3% execution accuracy on Spider <!-- cite:28 --> [[28]](references.md#ref-28). The general form of this idea is grammar- or regex-constrained decoding: Outlines reframes constrained generation as walking a finite-state machine derived from a regular expression or context-free grammar, masking the next-token distribution to admit only continuations the grammar allows, at negligible per-token overhead <!-- cite:29 --> [[29]](references.md#ref-29). Constrained decoding is how a code assistant guarantees that its output parses, returns valid JSON for a tool call, or conforms to a schema.

### <a id="sec-10.2"></a>10.2 Latency: Speculative Decoding and Caching

<a id="p-102-latency-speculative-decoding-and-caching-1"></a><!-- para:102-latency-speculative-decoding-and-caching-1 --> Autoregressive decoding emits one token per forward pass, which is the binding latency constraint for interactive code. **Speculative decoding** breaks the serial dependency: a small, cheap draft model proposes several tokens, and the large target model verifies them in a single parallel forward pass, accepting the longest correct prefix — producing identical output to the target model at a fraction of the wall-clock cost <!-- cite:27 --> [[27]](references.md#ref-27). Code assistants exploit a domain-specific twist: when applying an edit, most of the output overlaps existing code, so the *existing source itself* can serve as the draft, which is how Cursor's "fast apply" pushes a large model to roughly a thousand tokens per second <!-- cite:51 --> [[51]](references.md#ref-51). GitHub similarly reports cutting median completion latency from 1.4 seconds to 0.7 seconds with a FIM-trained custom model and speculative techniques <!-- cite:50 --> [[50]](references.md#ref-50). Caching is the other half: the suffix-prefix-middle FIM ordering exists partly so that appending tokens to the prefix does not invalidate the suffix's cached keys and values <!-- cite:5 --> [[5]](references.md#ref-5), and serving stacks cache the long repository-context prefix (prompt/KV caching) and the retrieval index (Section 11) so that only changed files are recomputed <!-- cite:50 --> [[50]](references.md#ref-50).

### <a id="sec-10.3"></a>10.3 Serving Shape: Autocomplete versus Agent

<a id="p-103-serving-shape-autocomplete-versus-agent-1"></a><!-- para:103-serving-shape-autocomplete-versus-agent-1 --> The serving target dictates the engineering, as Section 5 anticipated. **Autocomplete** is latency-bound: it favors smaller models, FIM, multi-query or grouped attention for fast batched inference (StarCoder chose multi-query attention precisely for this <!-- cite:8 --> [[8]](references.md#ref-8)), aggressive caching, and speculative decoding, all to hit a sub-second budget. **Agents** (Section 12) are throughput- and cost-bound: a single task issues many model calls over large contexts, so the relevant levers are total tokens, context-window cost, and parallel sampling, not first-token latency. **Quantization** (reduced-precision weights and activations) is the general throughput-and-memory lever applicable to both, though the published code-model reports treat it as standard infrastructure rather than a code-specific contribution. The cost consequences of these choices are quantified in Section 14.
