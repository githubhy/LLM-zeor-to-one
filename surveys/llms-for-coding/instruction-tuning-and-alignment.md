## 8 Instruction Tuning and Alignment

<a id="p-8-instruction-tuning-and-alignment-1"></a><!-- para:8-instruction-tuning-and-alignment-1 --> A pretrained code model can continue code, but it cannot reliably *do what it is asked*. Alignment closes that gap in two moves: supervised instruction tuning teaches the format of following requests, and preference optimization or reinforcement learning sharpens quality. Code has a structural advantage here that general LLMs lack — its reward can be *computed by execution* rather than judged by humans (Section 2) — and this advantage is the through-line of the section.

<!-- sec:8.1 -->
### <a id="sec-8.1"></a>8.1 Instruction Tuning: Teaching Models to Follow Code Requests

<a id="p-81-instruction-tuning-teaching-models-to-follow-code-requests-1"></a><!-- para:81-instruction-tuning-teaching-models-to-follow-code-requests-1 --> The cheapest path to an instruction-following code model is to generate instruction data with another model. Self-Instruct established the template: bootstrap a large instruction set from a small seed pool by prompting a strong model, then fine-tune on the result <!-- cite:15 --> [[15]](references.md#ref-15). Code Alpaca applied this directly to code (~20k samples). Two refinements proved especially effective for code:

- <a id="p-81-instruction-tuning-teaching-models-to-follow-code-requests-2"></a><!-- para:81-instruction-tuning-teaching-models-to-follow-code-requests-2 --> **Evol-Instruct (WizardCoder).** Rather than sampling new instructions flatly, WizardCoder iteratively *evolves* seed instructions into harder ones — adding constraints, increasing reasoning steps, requesting higher time/space complexity, or injecting buggy reference code as misdirection — then fine-tunes on the accumulated set <!-- cite:16 --> [[16]](references.md#ref-16). Starting from StarCoder-15B (33.6% HumanEval pass@1), WizardCoder-15B reaches 57.3% HumanEval and 51.8% MBPP; built on Code Llama-34B it reaches 71.5% / 61.2%, with the 15B model surpassing contemporaneous Claude and Bard numbers on HumanEval <!-- cite:16 --> [[16]](references.md#ref-16).
- **OSS-Instruct (Magicoder).** Instead of evolving model-invented instructions, Magicoder grounds generation in *real* open-source code: it prompts a model with 80k seed snippets drawn from filtered Stack data to produce 75k realistic instruction-response pairs, reducing the systematic bias of purely synthetic data <!-- cite:17 --> [[17]](references.md#ref-17). On Code Llama-Python-7B (37.8% HumanEval pass@1), Magicoder-CL reaches 60.4% and the combined MagicoderS-CL (OSS-Instruct plus Evol-Instruct, which are orthogonal) reaches 70.7% — on par with the contemporaneous ChatGPT number — while MagicoderS-DS on DeepSeek-Coder-6.7B reaches 76.8% with eight times fewer fine-tuning tokens than the official instruct model <!-- cite:17 --> [[17]](references.md#ref-17).

<a id="p-81-instruction-tuning-teaching-models-to-follow-code-requests-3"></a><!-- para:81-instruction-tuning-teaching-models-to-follow-code-requests-3 --> The pattern across both: small, *carefully constructed* instruction sets beat large generic ones — the alignment-stage echo of the data-quality thesis from Section 6.

### <a id="sec-8.2"></a>8.2 Preference Optimization and the Verifiable-Reward Advantage

<a id="p-82-preference-optimization-and-the-verifiable-reward-advantage-1"></a><!-- para:82-preference-optimization-and-the-verifiable-reward-advantage-1 --> Instruction tuning teaches format; preference optimization teaches *which* of two answers is better. The general-LLM workhorse is RLHF: train a reward model on human preference judgments, then optimize the policy against it with PPO. Direct Preference Optimization (DPO) simplifies this substantially — it shows the RLHF objective can be optimized with "only a simple classification loss," a binary-cross-entropy objective that fits an implicit reward model whose optimal policy is available in closed form, eliminating the separate reward model and the unstable online RL loop <!-- cite:18 --> [[18]](references.md#ref-18). Qwen2.5-Coder uses exactly DPO in its alignment stage <!-- cite:11 --> [[11]](references.md#ref-11).

<a id="p-82-preference-optimization-and-the-verifiable-reward-advantage-2"></a><!-- para:82-preference-optimization-and-the-verifiable-reward-advantage-2 --> The deeper point for code is *where the preference signal comes from*. Human-preference RLHF approximates a subjective, noisy, gameable target; a policy can reward-hack the learned reward model. For code, the preference can instead be drawn from **execution**: the chosen response passes the tests, the rejected one fails. This is **reinforcement learning with verifiable rewards (RLVR)** — the reward is an executable oracle (compiler plus test suite), so it is deterministic, cheap to scale, and cannot be gamed by exploiting a learned model. Qwen2.5-Coder's DPO preferences come from a code sandbox rather than human labels <!-- cite:11 --> [[11]](references.md#ref-11), and the reasoning-model results of Section 9 rest entirely on this property.

### <a id="sec-8.3"></a>8.3 Reinforcement Learning from Execution Feedback

<a id="p-83-reinforcement-learning-from-execution-feedback-1"></a><!-- para:83-reinforcement-learning-from-execution-feedback-1 --> The most explicit use of the executable reward is RL directly on test outcomes. CodeRL casts program synthesis as actor-critic RL: the pretrained model is the policy, generated programs are actions, and the compiler/test environment returns a terminal reward <!-- cite:19 --> [[19]](references.md#ref-19)

<a id="eq-1"></a><!-- eq:8-1 -->
$$
r(\hat{W}) = \begin{cases} -1.0 & \text{program fails to compile} \\ -0.6 & \text{runtime error during testing} \\ -0.3 & \text{compiles and runs but fails a unit test} \\ +1.0 & \text{passes all unit tests} \end{cases} \tag{1}
$$

<a id="p-83-reinforcement-learning-from-execution-feedback-2"></a><!-- para:83-reinforcement-learning-from-execution-feedback-2 --> A separate critic is trained to predict one of four outcomes — compile error, runtime error, failed test, passed test — and its token-level estimates turn the sparse terminal reward of Equation <!-- ref:8-1 -->[(1)](#eq-1) into intermediate, per-token returns, optimized with a self-critical baseline that weights by the relative return against a baseline program <!-- cite:19 --> [[19]](references.md#ref-19). CodeRL set new results on APPS and transferred zero-shot to MBPP <!-- cite:19 --> [[19]](references.md#ref-19).

<a id="p-83-reinforcement-learning-from-execution-feedback-3"></a><!-- para:83-reinforcement-learning-from-execution-feedback-3 --> The modern descendant is RLEF, which grounds a model in *multi-turn* execution feedback: the model proposes code, runs it, reads the failures, and revises across an episode, with the reward tied to test pass/fail <!-- cite:20 --> [[20]](references.md#ref-20). On the CodeContests competitive-programming benchmark, RLEF achieves new state-of-the-art results at both 8B and 70B scale while reducing the samples required by an order of magnitude; the 70B model improves from 37.5 to 40.4 on validation and reaches 41.2 on test (versus 38.0 with feedback limited to public tests), outperforming prior approaches such as AlphaCode and PPO-tuned Code Llama <!-- cite:20 --> [[20]](references.md#ref-20). RLEF is the bridge from Section 8 to Section 9: once a model learns to use execution feedback during training, using it at *inference* time — self-debugging, reflection, sampling-and-selection — becomes the next lever.
