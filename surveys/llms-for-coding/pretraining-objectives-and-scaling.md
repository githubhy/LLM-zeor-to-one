## 7 Pretraining Objectives, Tokenization, and Scaling

<a id="p-7-pretraining-objectives-tokenization-and-scaling-1"></a><!-- para:7-pretraining-objectives-tokenization-and-scaling-1 --> Section 4 introduced the autoregressive objective and the fill-in-the-middle transform in the abstract. This section is the engineering reality: how FIM is actually configured in production code models, how their tokenizers are built, how repositories are packed into training sequences, and what is known — and not known — about scaling code models.

<!-- sec:7.1 -->
### <a id="sec-7.1"></a>7.1 Fill-in-the-Middle in Production

<a id="p-71-fill-in-the-middle-in-production-1"></a><!-- para:71-fill-in-the-middle-in-production-1 --> The canonical FIM study (Section 4) recommends a 50% FIM rate, character-level spans, and joint training on the prefix-suffix-middle (PSM) and suffix-prefix-middle (SPM) orderings <!-- cite:5 --> [[5]](references.md#ref-5). Production code models adopt these settings with remarkable consistency. StarCoder applies FIM at the character level with a rate of 0.5, mixing PSM and an SPM variant equally <!-- cite:8 --> [[8]](references.md#ref-8). DeepSeek-Coder uses a 0.5 FIM rate in PSM mode applied at the document level before packing, and ablates 0%, 50%, and 100% FIM rates plus a masked-span-prediction alternative, settling on 50% PSM as the best trade between infilling and left-to-right quality <!-- cite:10 --> [[10]](references.md#ref-10). Code Llama applies FIM with probability 0.9, half PSM and half SPM, and notably suppresses the leading space that its SentencePiece tokenizer would otherwise insert, to limit the train/inference distribution shift between ordinary generation and infilling <!-- cite:6 --> [[6]](references.md#ref-6). Qwen2.5-Coder carries FIM into the repository-level stage with dedicated sentinel tokens <!-- cite:11 --> [[11]](references.md#ref-11). The lesson is that FIM is no longer a research option but a default, configured within a narrow band the field has converged on.

<!-- sec:7.2 -->
### <a id="sec-7.2"></a>7.2 Tokenizer Design at Scale

<a id="p-72-tokenizer-design-at-scale-1"></a><!-- para:72-tokenizer-design-at-scale-1 --> Code tokenizers are byte-level BPE with a digit-splitter and GPT-2-style pre-tokenization, sized to the code-plus-text mixture. StarCoder and StarCoder 2 use a 49,152-token vocabulary (including FIM sentinels) and report that raising it to 100k "did not improve performance," so they kept it small <!-- cite:9 --> [[9]](references.md#ref-9). DeepSeek-Coder uses 32,000 tokens <!-- cite:10 --> [[10]](references.md#ref-10); Qwen2.5-Coder inherits Qwen2.5's 151,646-token vocabulary and adds code/FIM special tokens <!-- cite:11 --> [[11]](references.md#ref-11). The recurring code-specific moves — whitespace-run tokens (Section 4), cross-whitespace merges, digit splitting — all target code's redundancy and numeric literals. One caveat for the careful reader: the published reports give vocabulary sizes and pre-tokenization rules but generally not per-language *fertility* tables (tokens per byte), so cross-model tokenizer-efficiency comparisons should be made qualitatively rather than from a precise published number.

<!-- sec:7.3 -->
### <a id="sec-7.3"></a>7.3 Repository-Level Pretraining and Long Context

<a id="p-73-repository-level-pretraining-and-long-context-1"></a><!-- para:73-repository-level-pretraining-and-long-context-1 --> Function-level pretraining teaches local syntax; repository-level pretraining teaches cross-file structure — that a symbol is defined in one file and used in another. The two published strategies differ instructively. DeepSeek-Coder performs dependency-aware ordering: it parses import statements (`import` in Python, `using` in C#, `#include` in C) to build a file dependency graph, runs a topological sort, and concatenates files so that a file appears after its dependencies, prepending each with a path comment <!-- cite:10 --> [[10]](references.md#ref-10). StarCoder 2 instead concatenates each repository's files in *random* order, delimited by a file-separator token, treating cross-file learning as a consequence of co-occurrence rather than explicit ordering <!-- cite:9 --> [[9]](references.md#ref-9). Both pair repository packing with longer contexts: StarCoder uses 8k, StarCoder 2 extends a 4k base to 16k via continued pretraining with a sliding window and increased RoPE base frequency, DeepSeek-Coder trains at 16k, and Qwen2.5-Coder extends to 128k via length extrapolation (YARN) in its repository stage <!-- cite:8 --> [[8]](references.md#ref-8), <!-- cite:9 --> [[9]](references.md#ref-9), <!-- cite:10 --> [[10]](references.md#ref-10), <!-- cite:11 --> [[11]](references.md#ref-11). Long context and retrieval are complementary answers to the same problem (Section 11): how to give the model the parts of a large repository that matter.

### <a id="sec-7.4"></a>7.4 Scaling and Data Mixture

<a id="p-74-scaling-and-data-mixture-1"></a><!-- para:74-scaling-and-data-mixture-1 --> Code models complicate the clean compute-optimal scaling story in two ways. First, the **mixture** matters and is counterintuitive. Qwen2.5-Coder ablates code:text:math ratios and finds that 70:20:10 outperforms higher-code mixtures — more code is not strictly better, because math and natural-language data improve code performance <!-- cite:11 --> [[11]](references.md#ref-11). DeepSeek-Coder uses 87% source code, 10% code-related English, and 3% other natural language <!-- cite:10 --> [[10]](references.md#ref-10). Second, deployment economics push models to **over-train** relative to the compute-optimal (Chinchilla) point: StarCoder 2 trains its 3B, 7B, and 15B models on 3.3-to-4.3 trillion tokens — far past the token count a compute-optimal law would prescribe — because the goal is inference efficiency, not training efficiency <!-- cite:9 --> [[9]](references.md#ref-9). Notably, none of these reports fit a code-specific scaling law with explicit exponents; the strongest scaling claim in the corpus is phi-1's qualitative one (Section 6), that data quality changes the shape of the curve rather than just shifting along it <!-- cite:12 --> [[12]](references.md#ref-12). The honest state of the art is that code scaling is governed by data quality and mixture at least as much as by the parameter-token frontier, and the precise functional form remains an open question (Section 18).
