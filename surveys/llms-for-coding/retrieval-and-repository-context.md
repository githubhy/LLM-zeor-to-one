## 11 Retrieval and Repository Context

<a id="p-11-retrieval-and-repository-context-1"></a><!-- para:11-retrieval-and-repository-context-1 --> Real software lives in repositories, not single files: the symbol you need is defined three directories away, the API you must call is documented in a sibling module. A model with even a large context window cannot hold an entire codebase, so the central problem of practical code assistance is *context construction* — selecting the parts of a repository that matter for the edit at hand. This section covers retrieval, the long-context alternative, and the tradeoff between them.

<!-- sec:11.1 -->
### <a id="sec-11.1"></a>11.1 Repository-Level Completion by Retrieval

<a id="p-111-repository-level-completion-by-retrieval-1"></a><!-- para:111-repository-level-completion-by-retrieval-1 --> The canonical method is RepoCoder, which pairs a similarity-based retriever with a code model in an *iterative* retrieve-generate loop <!-- cite:30 --> [[30]](references.md#ref-30). A retrieval database is built by sliding a window over the repository's files; the unfinished code at the cursor is the initial query. The key insight is that the unfinished code is a *poor* proxy for what the completion will need, so RepoCoder augments the query with its own previously generated completion and retrieves again: at iteration $i$ the retrieval uses the draft $\hat{Y}^{i-1}$ from iteration $i-1$, progressively closing the gap between "what I have written" and "what I am about to write" <!-- cite:30 --> [[30]](references.md#ref-30). Strikingly, a simple sparse retriever — bag-of-tokens snippets ranked by Jaccard similarity — performs on par with a dense neural retriever for this task <!-- cite:30 --> [[30]](references.md#ref-30). On the RepoEval benchmark (line, API-invocation, and function-body completion across real repositories), RepoCoder improves an in-file baseline by over 10% exact match and over 8% edit similarity, and consistently beats single-shot retrieval-augmented generation <!-- cite:30 --> [[30]](references.md#ref-30).

<!-- sec:11.2 -->
### <a id="sec-11.2"></a>11.2 Code Retrieval and Embeddings

<a id="p-112-code-retrieval-and-embeddings-1"></a><!-- para:112-code-retrieval-and-embeddings-1 --> Retrieval quality depends on how code is represented. Sparse lexical retrieval (as in RepoCoder) is a strong, dependency-free baseline, but dense retrieval uses learned code embeddings. The representation-model line of Section 3 supplies these: CodeBERT produces bimodal code/text embeddings for search <!-- cite:2 --> [[2]](references.md#ref-2), and UniXcoder unifies encoder, decoder, and encoder-decoder behavior with cross-modal pretraining (using AST and code-comment signals) to produce representations for code search and retrieval <!-- cite:14 --> [[14]](references.md#ref-14). In an assistant, these embeddings index repository chunks so that a natural-language request or a code context can pull semantically related code rather than only lexically overlapping code — complementary to the sparse approach.

<!-- sec:11.3 -->
### <a id="sec-11.3"></a>11.3 Long Context versus Retrieval

<a id="p-113-long-context-versus-retrieval-1"></a><!-- para:113-long-context-versus-retrieval-1 --> There are two ways to give a model repository-scale information: *retrieve* the relevant pieces (this section) or *extend the context window* to hold more of the repository directly (Section 7's long-context training, e.g., Qwen2.5-Coder's 128k window via length extrapolation <!-- cite:11 --> [[11]](references.md#ref-11), Code Llama's 100k extension <!-- cite:6 --> [[6]](references.md#ref-6)). These are complementary, not competing. Long context simplifies the pipeline and preserves exact text but pays a quadratic-ish attention cost and still cannot hold a large monorepo; retrieval scales to arbitrary repository size and is cheaper per query but risks omitting a needed snippet and adds index-maintenance complexity. The honest state of the field is that there is no published head-to-head settling which dominates for whole-repository coding; production systems generally combine them — retrieve candidate files, then place them in a long context — and the right mix is workload-dependent. Repository-level pretraining (Section 7) and retrieval are best understood as two attacks on the same problem the agentic systems of Section 12 confront directly: giving the model the right slice of a large codebase at the right moment.
