## 16 Safety, Security, Licensing, and Risk

<a id="p-16-safety-security-licensing-and-risk-1"></a><!-- para:16-safety-security-licensing-and-risk-1 --> Code models inherit the general risks of LLMs and add several that are specific to the modality: they can write insecure code that compiles and runs, they expand the attack surface when wired into autonomous agents, and they raise unresolved questions about the licensing of the code they were trained on. This section treats each in turn, grounded where possible in primary measurement.

<!-- sec:16.1 -->
### <a id="sec-16.1"></a>16.1 Insecure Code Generation

<a id="p-161-insecure-code-generation-1"></a><!-- para:161-insecure-code-generation-1 --> The foundational result is Pearce et al.'s "Asleep at the Keyboard," which evaluated GitHub Copilot across 89 scenarios spanning MITRE's top-25 weakness classes, producing 1,689 programs, and found approximately 40% to be vulnerable — 39.33% of the top-ranked suggestions specifically <!-- cite:41 --> [[41]](references.md#ref-41). The top suggestion matters most because users tend to accept the "best" completion, so a 40% vulnerability rate in first suggestions is a direct security concern, not an edge case <!-- cite:41 --> [[41]](references.md#ref-41). The hopeful counterpoint is that security is *steerable*. SVEN formulates "controlled code generation," using lightweight property-specific prefixes (continuous vectors, with the model's weights frozen) to steer generation toward secure or, adversarially, insecure code across 9 CWEs <!-- cite:42 --> [[42]](references.md#ref-42). On a 2.7B CodeGen model that generates secure code 59.1% of the time by default, SVEN's security hardening raises the secure rate to 92.3% while closely preserving functional correctness — and its adversarial mode degrades it to 36.8%, demonstrating that the same lever cuts both ways <!-- cite:42 --> [[42]](references.md#ref-42). Security can be substantially improved without retraining, but the dual-use nature of the technique is explicit.

<!-- sec:16.2 -->
### <a id="sec-16.2"></a>16.2 The Agentic Attack Surface

<a id="p-162-the-agentic-attack-surface-1"></a><!-- para:162-the-agentic-attack-surface-1 --> Autonomous coding agents (Section 12) introduce a risk class autocomplete does not: **indirect prompt injection**. An agent ingests untrusted repository content — issue titles, pull-request descriptions, code comments, commit messages, external documentation, and API responses — and can treat that content as instructions rather than data <!-- cite:52 --> [[52]](references.md#ref-52). The consequence is amplified by the agent's permissions: an agent granted broad workflow access (the ability to write files, commit, or mint tokens) turns a successful injection into repository tampering or credential theft, and industry security tracking reports prompt injection as the dominant driver of agentic-AI security failures in production through 2025–2026 <!-- cite:52 --> [[52]](references.md#ref-52). The mitigation principles mirror ordinary least-privilege security — scope the agent's permissions tightly, treat all repository content as untrusted input, and gate destructive actions — but the threat is structural to the observe-act loop and remains an active problem (Section 18).

### <a id="sec-16.3"></a>16.3 Licensing, Copyright, and Memorization

<a id="p-163-licensing-copyright-and-memorization-1"></a><!-- para:163-licensing-copyright-and-memorization-1 --> Because code carries licenses, training on it raises legal questions prose does not. The bellwether is the Doe v. GitHub class action: filed in 2022 with 22 claims, it was largely narrowed by a 2024 ruling that dismissed most claims, leaving primarily an open-source-license-violation claim and a breach-of-contract claim, with the matter proceeding on appeal <!-- cite:53 --> [[53]](references.md#ref-53). Notably, the DMCA copyright-management-information claim was dismissed on the finding that the tool "rarely memorized code" and did not reproduce plaintiffs' work identically — which ties the *legal* question directly to the *technical* one of verbatim memorization: the strength of a copyright claim depends on demonstrable regurgitation <!-- cite:53 --> [[53]](references.md#ref-53). The open ecosystem's answer is licensing-by-construction (Section 6): The Stack selects only permissively licensed files and offers a developer opt-out, an explicit attempt to make training data legally defensible rather than litigating after the fact <!-- cite:7 --> [[7]](references.md#ref-7), <!-- cite:8 --> [[8]](references.md#ref-8), <!-- cite:9 --> [[9]](references.md#ref-9). The licensing status of code models trained indiscriminately on copyleft code remains genuinely unsettled law, and memorization measurement for code is an underdeveloped area — a gap this survey flags rather than papers over.
