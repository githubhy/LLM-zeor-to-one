## 2 Scope and the Code Modality

<a id="p-2-scope-and-the-code-modality-1"></a><!-- para:2-scope-and-the-code-modality-1 --> Source code is, at the level of bytes, just another sequence of text — and the dominant code models are ordinary autoregressive Transformers trained with the same next-token objective used for natural language. So why does "LLMs for code" warrant a survey distinct from LLMs in general? Because code differs from prose in four ways that reshape every stage of the pipeline, from data curation to evaluation. This section makes that argument precise and fixes the vocabulary used throughout the survey.

<!-- sec:2.1 -->
### <a id="sec-2.1"></a>2.1 What Makes Code a Distinct Modality

<a id="p-21-what-makes-code-a-distinct-modality-1"></a><!-- para:21-what-makes-code-a-distinct-modality-1 --> **Formal syntax and executability.** Unlike natural language, code obeys a formal grammar and can be *run*. Codex operationalizes this by judging a generated program correct if and only if it passes a set of unit tests, and by executing untrusted model output inside a hardened sandbox (a gVisor container) to prevent a malicious or buggy sample from damaging the host <!-- cite:1 --> [[1]](references.md#ref-1). Executability is the hinge on which the other distinctions turn: a program's behavior, not merely its surface form, is observable.

<a id="p-21-what-makes-code-a-distinct-modality-2"></a><!-- para:21-what-makes-code-a-distinct-modality-2 --> **Functional correctness as verifiable ground truth.** Because code can be executed, it admits an *objective, automatable* correctness signal — pass/fail on tests — that natural-language generation lacks. Codex argues that match-based metrics such as exact match and BLEU are fundamentally deficient for code: they "are unable to account for the large and complex space of programs functionally equivalent to a reference solution," and the paper shows empirically that functionally *inequivalent* programs often score *higher* BLEU than equivalent ones <!-- cite:1 --> [[1]](references.md#ref-1). This is the single most consequential property of the modality. It is why evaluation is execution-based (Section 13), why reinforcement learning for code can use a cheap, ungameable reward (Section 8), and why test-time methods can verify their own outputs (Section 9). The deep learning community rarely gets a free, dense, correct reward function; for code, the compiler and test suite provide one.

<a id="p-21-what-makes-code-a-distinct-modality-3"></a><!-- para:21-what-makes-code-a-distinct-modality-3 --> **Bidirectional, long-range structure.** Prose is typically consumed left to right, but "code is seldom written in a single left-to-right pass and is instead repeatedly edited and refined" <!-- cite:4 --> [[4]](references.md#ref-4). An edit site has committed code on *both* sides, which motivates infilling objectives that condition on a right-hand suffix as well as a left-hand prefix — the fill-in-the-middle (FIM) capability developed in Section 4 and Section 7. Code also carries long-range dependencies (a function called far from its definition; a symbol imported from another file), which is why repository-scale context and retrieval (Section 11) and long-context training (Section 7) are first-class concerns.

<a id="p-21-what-makes-code-a-distinct-modality-4"></a><!-- para:21-what-makes-code-a-distinct-modality-4 --> **Distinctive tokenization pressure.** Code is dense with significant whitespace, indentation, and long compound identifiers. A naive text tokenizer wastes capacity on these; code models therefore add whitespace-run tokens (CodeGen extends the GPT-2 byte-pair vocabulary with tokens for repeated tabs and spaces <!-- cite:3 --> [[3]](references.md#ref-3)) or allow byte-level merges to cross whitespace (InCoder reports a 45% reduction in tokens needed to encode its corpus relative to GPT-2's tokenizer <!-- cite:4 --> [[4]](references.md#ref-4)). Tokenization is treated in Section 7.

### <a id="sec-2.2"></a>2.2 A Taxonomy of Code Tasks

<a id="p-22-a-taxonomy-of-code-tasks-1"></a><!-- para:22-a-taxonomy-of-code-tasks-1 --> "LLMs for code" spans a family of tasks, each grounded in the literature this survey cites:

- <a id="p-22-a-taxonomy-of-code-tasks-2"></a><!-- para:22-a-taxonomy-of-code-tasks-2 --> **Completion** — left-to-right continuation of partial code; the canonical IDE autocomplete setting <!-- cite:1 --> [[1]](references.md#ref-1), <!-- cite:3 --> [[3]](references.md#ref-3).
- **Natural-language-to-code synthesis** — generating a function from a docstring or specification, as in HumanEval <!-- cite:1 --> [[1]](references.md#ref-1) and CodeGen's multi-turn program synthesis <!-- cite:3 --> [[3]](references.md#ref-3).
- **Infilling / fill-in-the-middle** — generating a span given both its left and right context <!-- cite:4 --> [[4]](references.md#ref-4), <!-- cite:5 --> [[5]](references.md#ref-5), <!-- cite:6 --> [[6]](references.md#ref-6).
- **Program repair and editing** — fixing or transforming existing code, which InCoder frames as masking-and-infilling <!-- cite:4 --> [[4]](references.md#ref-4) and which the self-debugging line of work (Section 9) automates.
- **Code translation** — porting between languages, a functional-correctness task Codex cites as motivation <!-- cite:1 --> [[1]](references.md#ref-1).
- **Test generation** — synthesizing unit tests, the dual of the execution-based evaluation paradigm.
- **Repository-level and agentic editing** — coordinating multi-file changes against a real codebase and its test suite, the setting of SWE-bench (Section 13) and coding agents (Section 12).
- **Code understanding and retrieval** — non-generative tasks such as code search and summarization, the province of encoder models like CodeBERT <!-- cite:2 --> [[2]](references.md#ref-2) and UniXcoder <!-- cite:14 --> [[14]](references.md#ref-14).

### <a id="sec-2.3"></a>2.3 Scope of This Survey

<a id="p-23-scope-of-this-survey-1"></a><!-- para:23-scope-of-this-survey-1 --> This survey covers the end-to-end stack for code: the historical arc (Section 3) and first-principles fundamentals (Section 4); the training pipeline from data through alignment (Sections 5–9); inference, retrieval, and agents (Sections 10–12); evaluation (Section 13); and the practical and societal layer — tradeoffs, state of the art, safety, design guidance, and open problems (Sections 14–18). It centers on publicly documented methods and is not a vendor product comparison; proprietary system internals are discussed only where their developers have published them. For a complementary treatment focused on software-engineering practice rather than model internals, see the surveys by Jiang et al. <!-- cite:44 --> [[44]](references.md#ref-44) and Fan et al. <!-- cite:45 --> [[45]](references.md#ref-45), which this work cites for orientation and roadmap.
