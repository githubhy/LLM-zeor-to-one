# Evidence: Model Editing (ROME / MEMIT / Knowledge-Editing Critique)

Status: in progress. File-first evidence ledger for the mechanistic-interpretability survey. Each block below answers one question with source-traceable claims.

## Q1: ROME (Meng et al. 2022, arXiv:2202.05262) — causal tracing localizes facts to mid-layer MLPs; the rank-one update to W_proj as a constrained least-squares (key-value memory) edit. Give the closed-form update.

- **Finding:** ROME treats a transformer MLP's second (down-projection) weight matrix $W_{proj}$ (called $W_{out}$/$W_{fc2}$ in some notations) as a linear associative (key-value) memory. Causal tracing (a corrupt-then-restore activation-patching sweep) shows that factual recall at the **last subject token**, in a band of **mid-network MLP layers**, has a much larger causal effect on the model's output than attention modules at the same position — this localizes "where the fact lives" and motivates editing exactly that site. ROME then edits one fact by solving a rank-one-constrained least-squares problem: find the minimal-norm perturbation to $W_{proj}$ that forces the key vector for the new subject to map exactly to a target value vector encoding the new object, while leaving the memory's response to all other keys as close to unchanged as possible.
- **Mechanism / derivation notes:**
  1. **MLP-as-memory setup.** For an MLP layer $l$, first layer $W_{fc}$ produces a key-like intermediate; ROME's operative object is the second layer $W_{proj}^{(l)}$ such that (schematically) $m^{(l)} = W_{proj}^{(l)} k$, where $k$ is the (nonlinear) hidden activation at the subject's last token position and $m^{(l)}$ is the MLP's output contribution added into the residual stream. The paper frames $W_{proj}$ as an associative memory storing many key→value pairs implicitly (à la Kohonen/linear associative memory: $Wk_i \approx v_i$ for many stored $(k_i, v_i)$).
  2. **Key $k_*$**: computed by running the *subject* text (e.g., "The Eiffel Tower is located in") through the network and taking the MLP's input activation at the last subject token, optionally averaged over several text prefixes for robustness (this is the "key selection" step — average of $k$ over a small set of random prefixes prepended to the subject, to make the key robust to context).
  3. **Value $v_*$**: found by a separate optimization — a gradient-descent search over a single vector $z$ (equivalently, the target output of layer $l$ at the subject token) that, when substituted into the forward pass at layer $l$, maximizes the probability the model assigns to the new target object $o^*$ (subject to a KL-divergence regularizer against the original next-token distribution on unrelated text, to preserve fluency/essence). This optimized vector is the desired new $v_*$.
  4. **Constrained least-squares insert.** Given $(k_*, v_*)$, ROME seeks the new weight matrix $\hat W$ that (a) satisfies the hard constraint $\hat W k_* = v_*$ exactly, and (b) among all matrices satisfying that constraint, minimizes $\lVert \hat W k_* ... \rVert$ interference with the memory's existing key→value mappings, formalized via a second-moment/covariance statistic $C = K K^\top$ (estimated over a large sample of real key vectors $k_i$ drawn from a corpus, e.g. Wikipedia text run through the model) representing "the set of keys the memory must continue answering correctly for."
  5. **Closed-form solution.** The constrained-least-squares problem has the classical closed-form rank-one update (structurally identical to a Sherman–Morrison-style / Lagrange-multiplier solution for a linear-associative-memory edit):
     $$\hat W = W + \Lambda (C^{-1}k_*)^\top, \qquad \Lambda = \frac{v_* - W k_*}{(C^{-1}k_*)^\top k_*}$$

     Here $\Lambda$ is a column vector (the "residual" that must be added, scaled so that the constraint $\hat W k_* = v_*$ holds exactly), $C^{-1}k_*$ is the direction of the rank-one update (whitened by the inverse covariance so the update is minimally disruptive to other stored keys), and the denominator normalizes so the constraint is satisfied. This is a **rank-one update** to $W$ (outer product of two vectors), hence "ROME" = Rank-One Model Editing.
  6. **Causal tracing mechanism (the localization step, prior to editing).** Run the network three times per prompt: (i) clean run (records all hidden states); (ii) corrupted run (subject-token embeddings perturbed with noise, degrading the prediction); (iii) corrupted-with-restoration run (subject corrupted, but one hidden state — an individual MLP or attention output at a given layer/token — is patched back to its clean value). The gap in output probability of the correct answer between (ii) and (iii), averaged over many facts, is the **Average Indirect Effect (AIE)** of that hidden state. Sweeping over all (layer, token-position) pairs and separately over "MLP-only" vs "attention-only" restoration produces the causal-tracing heatmaps that show a sharp AIE peak at mid-layer MLPs, last-subject-token position.
- **Values:**
  - Rank-one update formula: $\hat W = W + \Lambda(C^{-1}k_*)^\top$, $\Lambda = (v_* - Wk_*)/((C^{-1}k_*)^\top k_*)$ — confirmed via multiple independent secondary summaries (EmergentMind topic page, GitHub README paraphrase) though the exact primary-source equation number could not be directly quoted verbatim (WebFetch on the PDF returned only encrypted/binary stream content, not extractable text — see Gaps below).
  - Causal tracing peak: **AIE ≈ 8.7% at layer 15** for GPT-2-XL (1.5B params), with **total ATE (Average Treatment Effect) ≈ 18.6%** — i.e. mid-layer MLP restoration recovers roughly half the corrupted-run's probability deficit.   - Models: GPT-2-XL (1.5B params) and GPT-J (6B params, 28 layers) are the two primary evaluation models in the paper.

## Q2: MEMIT (Meng et al. 2023, "Mass-Editing Memory in a Transformer," arXiv:2210.07229) — editing thousands of facts by spreading the update across a range of MLP layers; how it scales beyond ROME.

- **Finding:** MEMIT (ICLR 2023) generalizes ROME from a single rank-one edit at one MLP layer to a **batched, multi-layer** edit that inserts thousands of facts simultaneously. Instead of forcing the entire key→value change to be absorbed by one layer's weight matrix (which becomes numerically/semantically destructive once many facts are packed into one rank-one-per-fact update), MEMIT distributes the required change in the residual stream evenly across a *range* of critical mid-layer MLPs identified by causal tracing, and solves a batched least-squares insertion (many key/value pairs at once, not one) at each of those layers. This is what lets it scale to inserting up to ~10,000 associations into GPT-J (6B) / GPT-NeoX (20B) where naively repeating ROME's single-layer edit breaks the model far sooner.
- **Mechanism / derivation notes:**
  1. **Same per-fact value optimization as ROME.** For each fact $i$ to be inserted, MEMIT first (as in ROME) runs an optimization to find the target hidden state $z_i$ at the *final* critical layer $L$ (the last layer in the target range) that would make the model output the desired object — i.e. it solves for what the residual-stream state at layer $L$, subject-last-token position, "should" be to encode the new fact, via gradient descent + KL regularization to preserve fluency, exactly as ROME's value-vector search.
  2. **Residual delta and even spreading.** Let $h_i^L$ be the *actual* (unedited) hidden state at layer $L$ for fact $i$. The total required change is $\delta_i = z_i - h_i^L$. Rather than injecting all of $\delta_i$ at layer $L$ alone, MEMIT spreads it evenly across every layer $l$ in the critical range $[l_0, L]$: each layer receives a residual share $r_i^l = \delta_i / (L - l + 1)$ (i.e. the further back from $L$, the more layers share the same total delta, but the increment written at each layer is identical in this even-split scheme — equivalently, the layers accumulate the shared increments additively through the residual stream so that by the time the stream reaches layer $L$, the full $\delta_i$ has been added). This is the key departure from ROME: instead of one layer absorbing a full-magnitude, potentially large perturbation, several layers each absorb a small fraction, reducing per-layer interference with unrelated stored facts.
  3. **Batched constrained least-squares per layer.** At each critical layer $l$, MEMIT now has (for a batch of $u$ facts being inserted together) a matrix of keys $K_1^l$ (existing/old keys, used to estimate the preserved-memory covariance term, exactly as ROME's $C$) and a matrix of new keys/target residual-shares for the new facts. It solves a batched analogue of ROME's rank-one closed form — a multi-key-value least-squares problem whose solution is a **rank-$u$** (not rank-1) update to $W_{proj}^{(l)}$ that (a) satisfies the new facts' key→value(-share) constraints as closely as possible in a least-squares sense (relaxed rather than exact equality, since satisfying $u$ constraints simultaneously and exactly is generally infeasible for a single matrix) while (b) minimizing disruption to the existing covariance-weighted key set, using the same $C = KK^\top$ regularization device as ROME.
  4. **Critical-layer selection via causal tracing.** As in ROME, MEMIT first runs causal tracing to find which MLP layers causally mediate factual recall at the subject's last token; for GPT-J, this identifies a *range* (rather than ROME's single best layer) $R = \{3,4,5,6,7,8\}$ as the critical band whose causal effect is high and roughly comparable across the range, motivating spreading the edit across all of them rather than concentrating in just one.
  5. **Why spreading scales better.** The paper (and later replication work) frames the single-layer ROME edit as fragile at scale because forcing one weight matrix to satisfy an increasing number of rank-one constraints eventually corrupts its response to unrelated (unedited) inputs — this is the mechanism behind catastrophic forgetting under naive iterated ROME edits. Spreading the same total required change across several layers keeps each individual layer's perturbation smaller in norm, which empirically delays and softens this degradation.
- **Values:**
  - Scale claimed: MEMIT demonstrated editing **up to ~10,000 factual associations** in a single batched edit into GPT-J (6B) and GPT-NeoX (20B), "exceeding prior work by orders of magnitude" (prior single-edit methods including ROME were evaluated at the scale of one fact at a time or small batches). Condition: zsRE and CounterFact benchmarks, GPT-J-6B / GPT-NeoX-20B.
  - Critical layer range for GPT-J: $R = \{3, 4, 5, 6, 7, 8\}$ — "GPT-J has a concentration of mediating states with a causal role for a range of MLP modules, with this effect diminishing after layer 8" (paraphrased from search-result summary of the paper).
  - Residual-spreading formula (paraphrased/reconstructed from search snippets, not a verbatim primary quote): $r_i^l = \dfrac{z_i - h_i^L}{L - l + 1}$, i.e. the residual needed at the final critical layer $L$ is divided by the count of layers from $l$ to $L$ inclusive and added at each layer in the range.
  - Related downstream finding (not MEMIT's own paper, but directly relevant to "how it scales beyond ROME"): Gupta, Rao et al. (2024), "Model Editing at Scale leads to Gradual and Catastrophic Forgetting," ACL Findings 2024 / arXiv:2401.07453 — found empirically that under sequential/scaled editing, **MEMIT's catastrophic-forgetting onset occurs at roughly ~1,400 edits** and MEMIT "forgets almost three times fewer previously edited facts" than ROME at comparable scale, with a longer gradual-forgetting phase before the catastrophic transition. This is an independent replication/stress-test, not a MEMIT-paper-reported number.

## Q3: Knowledge-editing evaluation and critique — efficacy / generalization / specificity / fluency metrics; ripple effects (Cohen et al. 2023); locate-then-edit critique (Hase et al. 2023, "Does Localization Inform Editing?").

- **Finding:** Standard single-fact editing evaluation (as introduced/used by ROME's CounterFact benchmark) scores an edit on **efficacy** (did the edit take, on the literal edit prompt?), **generalization** (does it hold under paraphrases of the same prompt?), **specificity** (are unrelated "neighborhood" facts left unchanged?), plus **fluency** and **consistency** as quality/side-effect checks on generated text. Cohen et al. (2023) argue this local/neighborhood view is insufficient: a real-world fact edit has logical **ripple effects** on other facts entailed by or related to it, and their RippleEdits benchmark shows current editing methods (ROME, MEMIT, MEND) largely fail to propagate edits to these logically-entailed consequences, while a simple in-context-editing baseline does comparatively well. Separately, Hase et al. (2023) present a more fundamental critique of the "locate-then-edit" paradigm itself: they find that ROME's causal-tracing-based localization (finding *where* a fact's causal effect is highest) is **not predictive of where editing that fact actually works best** — editing succeeds about equally well when applied to layers far from the causally-traced site, meaning "successfully editing a fact via ROME at layer $L$" cannot be used as evidence that the fact was "stored at" layer $L$.
- **Mechanism / derivation notes:**
  1. **Efficacy / Generalization / Specificity (ROME/CounterFact convention).** Efficacy Score (ES): binary/probability that the post-edit model assigns higher probability to the new target object than the old one, evaluated on the exact edit prompt. Paraphrase/Generalization Score (PS): same check but on paraphrased rewordings of the edit prompt (tests whether the edit generalizes beyond surface form rather than only pattern-matching the literal string). Neighborhood Score (NS)/specificity: same check but on prompts about *different* subjects that share the same relation (e.g., other landmarks' locations) — the edit should NOT have changed these; a high NS means the edit didn't spill over. Fluency and Consistency are secondary generation-quality checks (fluency via measures like weighted n-gram entropy of generated continuations to catch repetition/degeneration; consistency via, e.g., cosine similarity between generated continuations and reference text about the new object, checking the edited model "talks about" the object coherently, not just answers one probe correctly).
  2. **Ripple-effect taxonomy (Cohen et al. 2023, RippleEdits).** Six evaluation criteria, split into two families: (a) *should change* — Logical Generalization (LG: does an edit propagate to logically entailed facts, e.g. changing a person's country of citizenship should change "does this person live in a country in Europe?"), Compositionality I (CI: prompts chaining the edited relation with a second relation about the *same* subject), Compositionality II (CII: prompts chaining a second relation about the *edited object* composed with the original relation), Subject Aliasing (SA: the edit should transfer to referring expressions/aliases of the same subject); (b) *should NOT change* — Forgetfulness/Preservation (FF: other attributes of the same subject untouched) and Relation Specificity (RS: same relation, different, unrelated subjects untouched — the classic specificity/neighborhood check generalized). The point of the taxonomy is that (a)-type criteria are exactly what single-fact "efficacy/generalization" scoring on CounterFact/zsRE misses — an edit can look "successful" by ES/PS/NS while still leaving the model's broader factual graph logically inconsistent.
  3. **Empirical result pattern.** Across RippleEdits, editing methods show large method × ripple-type interaction (no single method dominates all six criteria) and generally weaker performance on the logically-entailed criteria than on the narrow single-fact criteria: Subject Aliasing accuracy is consistently high, while performance on Logical Generalization / Compositionality is markedly lower and more variable across models/methods. A simple in-context-editing baseline (prepending the new fact as context rather than editing weights) scores best overall on the benchmark, which the authors read as evidence that current weight-editing methods under-propagate logical consequences relative to what an in-context / retrieval-style approach achieves "for free" via the model's ordinary in-context reasoning.
  4. **Hase et al.'s locate-then-edit critique — the logical structure of the argument.** The paradigm (ROME, MEMIT) is: (i) run causal tracing to find the layer(s) with highest causal/indirect effect on factual recall ("localization"); (ii) edit weights at that located layer, on the implicit assumption that the location that mediates recall is also the best/necessary location to intervene for a durable, side-effect-free edit. Hase et al. test this assumption directly by editing at *many* different layers (not just the causally-traced one) and measuring edit success (e.g., a "rewrite score") as a function of edit layer, then comparing how much of the variance in edit success is explained by (a) edit layer alone vs (b) edit layer **plus** the causal-tracing effect size at that layer.
  5. **Quantitative decomposition.** A regression of edit success on edit layer alone achieves $R^2 \approx 0.585$ (58.5%) on average across conditions — i.e. *which layer you choose to edit* is highly predictive of success (some layers are just better places to insert an edit, largely independent of the causal-tracing profile). Adding the causal-tracing effect size as an additional regressor improves $R^2$ by **at most 0.03** over the layer-only regression — i.e. knowing "how causally important this layer was for recall" adds almost nothing once you already know which layer you're editing. This is the empirical core of "does localization inform editing? — no": the located layer's causal importance is nearly orthogonal to how good an edit site it is.
  6. **Interpretation.** Causal tracing answers "where does perturbing/restoring this representation change the *output*, holding the rest of the (frozen) forward pass fixed" — a statement about the *existing* computation's sensitivity. Editing success answers "where can I overwrite weights and get a *new*, stable, generalizing association written in" — a statement about which layer's downstream circuitry can absorb and propagate a modified representation. Hase et al.'s point is these are different questions with different answers, so a locate-then-edit method's *editing success* should not be read as validating its *localization claim* (and vice versa) — the classic "editing success is not evidence for storage location" caution now widely cited in mechanistic-interpretability critiques of knowledge editing.
- **Values:**
  - RippleEdits benchmark size: **~5,000 factual-edit entries** ("RippleEdits is a diagnostic benchmark of 5K factual edits"), six evaluation criteria (LG, CI, CII, SA, FF, RS).
  - Subject Aliasing (SA) accuracy: **≥ 86.8** across all model/method settings tested (consistently the easiest criterion), vs. markedly lower and more variable scores on Logical Generalization / Compositionality criteria (exact per-method numeric table not independently confirmed from primary source — see Gaps).
  - Method comparison on RippleEdits: MEND outperforms ROME and MEMIT on Logical Generalization but underperforms them on Compositionality I and II — "different methods might better capture different types of ripple effects" (paraphrased from search-result summary); an in-context-editing (ICE) baseline obtains the best overall scores on the benchmark.
  - Hase et al. (2023) quantitative result: edit-layer-only regression $R^2 \approx 58.5\%$ average across conditions; adding causal-tracing effect size improves $R^2$ by **at most 0.03** — condition: NeurIPS 2023 paper's layer-vs-tracing regression analysis (exact model(s)/dataset for this specific regression not independently re-confirmed beyond the search snippet — see Gaps).
