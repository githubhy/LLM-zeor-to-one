## <a id="sec-2"></a>2 Fundamentals

<a id="p-2-fundamentals-1"></a><!-- para:2-fundamentals-1 --> Everything a multimodal LLM does rests on one trick performed well: turning a continuous signal into a short sequence of vectors that a token-based transformer can read, in a geometry the transformer already understands. This section builds that trick from the ground up. We start from the obstacle — a discrete-sequence machine fed a continuous field — then derive, in order, how an image becomes a sequence of patch tokens, how a vision encoder processes those tokens, how the encoder's features are aligned to language so they are *meaningful* to an LLM, and finally how the aligned features are handed to the LLM itself. The base transformer and the autoregressive objective are assumed; a reader who wants them from first principles will find them in the companion survey on [language models from first principles](../llms-for-coding/language-models-from-first-principles.md). The treatment here is at *headline* depth: each result is derived, motivated with intuition and a signal-processing analogy, illustrated with a worked numerical example, and accompanied by its complexity and failure modes. The heavier algebra — the mutual-information reading of the contrastive loss, the full encoder forward pass — is deferred to Appendices A and B, which this section forward-references at the relevant points.

<!-- sec:2.1 -->
### <a id="sec-2.1"></a>2.1 The obstacle: a discrete machine meets a continuous field

<a id="p-21-the-obstacle-a-discrete-machine-meets-a-continuous-field-1"></a><!-- para:21-the-obstacle-a-discrete-machine-meets-a-continuous-field-1 --> A transformer language model is a function on *sequences of integers*. Its input is a list of token ids $x_t \in \{1,\dots,|V_t|\}$; its very first operation is a table lookup that replaces each id with a learned row of an embedding matrix, producing a vector in $\mathbb{R}^{d}$. The model never sees characters — only indices and the vectors they point at. This is worth saying slowly because it is the crux of the whole field: the embedding table is the *only* place raw input enters the network, and it accepts exactly one kind of object, a discrete symbol from a finite alphabet.

<a id="p-21-the-obstacle-a-discrete-machine-meets-a-continuous-field-2"></a><!-- para:21-the-obstacle-a-discrete-machine-meets-a-continuous-field-2 --> An image is the opposite kind of object. It is a dense field of intensities $x_v \in \mathbb{R}^{H\times W\times C}$ — height $H$, width $W$, and $C$ colour channels — with no alphabet, no finite vocabulary, and no canonical left-to-right ordering. An audio waveform is the same story in one dimension: a continuous amplitude sampled densely in time. For a reader from signal processing the framing is immediate. The transformer is a *discrete-sequence processor*, like a digital filter that consumes one sample per clock tick; the image is a *continuous-domain signal*, like an analog waveform before the converter. You cannot wire one to the other directly. What sits between them is an interface that does for an image what an analog-to-digital front-end does for a waveform: it *partitions* the signal into pieces, *summarizes* each piece as a feature vector, and *lays the vectors out* as a finite sequence the downstream block can clock through. The three subsections that follow are exactly the three stages of that front-end — partition (§ 2.2), summarize (§ 2.3), and align to the downstream geometry (§ 2.4–2.5) — followed by the hand-off to the LLM (§ 2.6).

<a id="p-21-the-obstacle-a-discrete-machine-meets-a-continuous-field-3"></a><!-- para:21-the-obstacle-a-discrete-machine-meets-a-continuous-field-3 --> One subtlety distinguishes the image case from a textbook sampler. A waveform sampler produces scalar samples; we instead want each piece to become a rich *feature vector*, because a small image patch already contains structure (an edge, a texture, a fragment of a letter) that a single number cannot carry. So the "summarize" stage is learned, not fixed: the front-end is trained, and what it learns to extract is the subject of the rest of this section.

### <a id="sec-2.2"></a>2.2 Patch embedding: an image as a sequence of tokens

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-1"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-1 --> **Start with the picture in numbers.** Take a $224\times 224$ RGB image — the standard input size for the vision encoder CLIP <!-- cite:1 -->[[1]](#ref-1) uses. Choose a patch side of $P = 14$ pixels and tile the image into a non-overlapping grid of $14\times 14$ squares. The grid is $224/14 = 16$ patches across and $16$ down, so there are $N_v = 16\times 16 = 256$ patches. Each patch is a little $14\times 14\times 3$ block of pixels; flatten it and it is a vector of $14^2\cdot 3 = 588$ numbers. We now have $256$ vectors of length $588$ — but the transformer wants vectors of its own working width $d_v$ (for the ViT-Large encoder CLIP uses, $d_v = 1024$). A single learned linear map $\mathbf{E}\in\mathbb{R}^{588\times 1024}$ projects every flattened patch up to length $1024$. That is the whole patch-embedding step: $256$ patches in, $256$ token vectors of width $1024$ out. Prepend one extra learned vector (the "class" token, explained below) and the encoder receives a sequence of $257$ tokens — structurally identical to a sentence of $257$ words.

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-2"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-2 --> **The general statement.** This procedure is the Vision Transformer (ViT) of Dosovitskiy et al. <!-- cite:2 -->[[2]](#ref-2). Reshape the image $x_v\in\mathbb{R}^{H\times W\times C}$ into a sequence of $N_v = HW/P^2$ flattened patches $\mathbf{x}_p^k\in\mathbb{R}^{P^2 C}$, project each with a shared matrix $\mathbf{E}\in\mathbb{R}^{(P^2 C)\times d_v}$, prepend a learnable class token $\mathbf{x}_{\text{class}}$, and add a learned position embedding $\mathbf{E}_{\text{pos}}\in\mathbb{R}^{(N_v+1)\times d_v}$ to inject ordering information the bag of patches would otherwise lack:

<a id="eq-1"></a><!-- eq:2-1 -->
$$
\mathbf{z}_0 = \big[\, \mathbf{x}_{\text{class}};\ \mathbf{x}_p^1\mathbf{E};\ \mathbf{x}_p^2\mathbf{E};\ \cdots;\ \mathbf{x}_p^{N_v}\mathbf{E}\,\big] + \mathbf{E}_{\text{pos}} \tag{1}
$$

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-3"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-3 --> The result $\mathbf{z}_0\in\mathbb{R}^{(N_v+1)\times d_v}$ is the input sequence to the encoder. Equation <!-- ref:2-1 -->[(1)](#eq-1) is the entire bridge from "continuous field" to "token sequence."

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-4"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-4 --> **Intuition — patch embedding is a learned filterbank.** A signal-processing reader can see exactly what $\mathbf{E}$ is. Applying one fixed $14\times 14\times 3$ patch and projecting it with one row of $\mathbf{E}^\top$ is an inner product of the patch with a $588$-tap kernel — a 2-D convolution evaluated once per patch, with stride equal to the kernel size so the patches do not overlap. The $1024$ rows of $\mathbf{E}^\top$ are $1024$ such kernels: a *learned filterbank* applied to a strided tiling of the image, exactly the front-end of a multirate analysis bank, except the filters are trained end-to-end rather than designed. (This equivalence is literal: ViT's patch projection can be implemented as a single `Conv2d` with kernel size and stride both equal to $P$.) The position embedding $\mathbf{E}_{\text{pos}}$ then restores the spatial index the tiling threw away, the visual analogue of the timestamp a frame carries in a framed audio pipeline.

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-5"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-5 --> **The class token.** One token in the sequence carries no image content at its input — the prepended $\mathbf{x}_{\text{class}}$. Its job is to be a *readout register*: because self-attention lets every token gather information from every other, after $L$ layers the class token's slot has had the chance to pool a summary of the whole image, and its final-layer state is used as the single-vector image representation. We will see in § 2.4 that this one vector is what gets aligned to language. (It is also, as § 2.4 notes, a bottleneck: a whole image squeezed into one vector. Instruction-tuned VLMs instead keep the full grid of $N_v$ patch tokens.)

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-6"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-6 --> **Complexity — why resolution is the cost driver.** The patch count is quadratic in the linear resolution: $N_v = HW/P^2$. Doubling the image side quadruples $N_v$, and because the encoder's self-attention costs $O(N_v^2 d_v)$ per layer (every patch attends to every patch), the encoder cost grows with the *fourth power* of the image side at fixed patch size. Concretely, moving CLIP-ViT-L from $224$ px to $336$ px takes $N_v$ from $256$ to $(336/14)^2 = 576$, and the attention term grows by $(576/256)^2 \approx 5.1\times$. This single fact — tokens scale as resolution squared, attention as tokens squared — is the engine behind much of the architecture inventory later in this survey: token-reducing connectors (§ 3), dynamic-resolution schemes such as Qwen2-VL's <!-- cite:7 -->[[7]](#ref-7), and the perennial tension between seeing fine detail and paying for the tokens it costs.

<a id="p-22-patch-embedding-an-image-as-a-sequence-of-tokens-7"></a><!-- para:22-patch-embedding-an-image-as-a-sequence-of-tokens-7 --> **Failure mode and the encoder-free alternative.** Patchification is lossy in a specific way: structure *smaller* than a patch is summarized into one $588\to1024$ projection and can be washed out. Small text in a document, a thin chart gridline, a distant face — these live below the patch scale and are the first things a low-resolution VLM gets wrong, which is why document and chart benchmarks punish small inputs. There is also a more radical option: skip the separate, pretrained encoder entirely and feed raw, linearly projected patches straight into the LLM's first layer, letting the language model learn the embedding itself. This is the design of Fuyu <!-- cite:14 -->[[14]](#ref-14); it trades the strong visual prior of a pretrained encoder for architectural simplicity and native handling of arbitrary resolutions. We return to it in the architecture inventory; for now it marks the boundary of the "use an encoder" assumption that the rest of this section makes.

### <a id="sec-2.3"></a>2.3 The vision encoder: self-attention over patches

<a id="p-23-the-vision-encoder-self-attention-over-patches-1"></a><!-- para:23-the-vision-encoder-self-attention-over-patches-1 --> Once the image is a sequence $\mathbf{z}_0$ of patch tokens, processing it is *ordinary transformer machinery*. ViT applies a stack of $L$ identical pre-norm blocks, each a multi-head self-attention (MSA) sublayer and an MLP sublayer, both wrapped in layer normalization and a residual connection:

<a id="eq-2"></a><!-- eq:2-2 -->
$$
\begin{aligned}
\mathbf{z}'_\ell &= \mathrm{MSA}\big(\mathrm{LN}(\mathbf{z}_{\ell-1})\big) + \mathbf{z}_{\ell-1}, \\
\mathbf{z}_\ell &= \mathrm{MLP}\big(\mathrm{LN}(\mathbf{z}'_\ell)\big) + \mathbf{z}'_\ell,
\end{aligned}
\qquad \ell = 1,\dots,L \tag{2}
$$

<a id="p-23-the-vision-encoder-self-attention-over-patches-2"></a><!-- para:23-the-vision-encoder-self-attention-over-patches-2 --> with the final image representation read off the class token's slot, $\mathbf{y} = \mathrm{LN}(\mathbf{z}_L^0)$. The self-attention inside MSA is the same scaled-dot-product attention used in any language transformer — queries, keys, and values projected from each token, similarity scores softmaxed and used to mix the values — and we do not re-derive it here; the first-principles derivation lives in the companion [attention appendix](../llms-for-coding/appendix-a-qkv-first-principles.md), and the encoder-specific elaboration (multi-head bookkeeping, the GELU MLP, why pre-norm) is developed in Appendix A of this survey. The one conceptual difference from a *language* transformer is that the attention is **bidirectional**: a patch may attend both "left" and "right," because an image has no causal arrow of time the way generated text does. The vision encoder is an *encoder* in the strict sense — it reads the whole image at once — whereas the LLM it feeds is a causal *decoder*.

<a id="p-23-the-vision-encoder-self-attention-over-patches-3"></a><!-- para:23-the-vision-encoder-self-attention-over-patches-3 --> **Intuition — what the layers buy.** Patch embedding sees each $14\times 14$ square in isolation; a single edge fragment means nothing without its neighbours. Self-attention is the mechanism that lets patch tokens *exchange* information, so that by the upper layers a token's vector encodes not just its own patch but its role in the larger object — this corner belongs to a wheel, that texture is fur. In filterbank terms, patch embedding is the analysis front-end and the attention stack is a global, content-adaptive mixing network laid on top of it: unlike a convolution, which mixes only a fixed local neighbourhood, attention can route information between any two patches in one step, which is precisely the "much less image-specific inductive bias" Dosovitskiy et al. note as ViT's defining departure from convolutional networks <!-- cite:2 -->[[2]](#ref-2).

<a id="p-23-the-vision-encoder-self-attention-over-patches-4"></a><!-- para:23-the-vision-encoder-self-attention-over-patches-4 --> **Inductive bias and its consequence.** Because a ViT bakes in almost no 2-D prior — locality and translation-equivariance are not built into the layers, only the patch grid and the position embeddings carry spatial structure — it must *learn* spatial relationships from data, and so it is data-hungry: ViT's own ablations show it underperforms convolutional nets on mid-sized datasets and overtakes them only at large scale <!-- cite:2 -->[[2]](#ref-2). This is why the dominant practice is not to train the vision encoder from scratch per task but to *reuse* one pretrained at scale — and the most useful such pretraining aligns the encoder to language directly, which is the subject of the next subsection. It also explains a recurring deployment fact: because the position embeddings are learned for a specific token-grid size, feeding a different resolution requires *interpolating* them, and getting that interpolation right is a routine source of fine-tuning bugs.

### <a id="sec-2.4"></a>2.4 Contrastive alignment: teaching vision to speak the language's geometry

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-1"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-1 --> A vision encoder trained to classify ImageNet produces features good for ImageNet's $1000$ labels. But an LLM does not think in $1000$ labels; it thinks in the geometry of its text embeddings, where "a golden retriever" and "a dog" sit close and "a dog" and "a differential equation" sit far. For the encoder's output to be *meaningful* to the LLM, the two geometries must be aligned. The breakthrough that made this practical is **contrastive language–image pretraining (CLIP)** <!-- cite:1 -->[[1]](#ref-1): train the vision encoder and a text encoder *together* so that an image and its caption land near each other and an image and someone else's caption land far apart. The encoder that falls out has features already arranged by *meaning expressed in language* — which is exactly what a downstream LLM can use.

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-2"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-2 --> **The setup, in numbers first.** Sample a batch of $N$ image–caption pairs from the web — CLIP used $N = 32{,}768$. Encode each image to a vector and each caption to a vector, project both into a shared $d_e$-dimensional "joint embedding" space, and $L2$-normalize so each lives on the unit sphere. Now form the $N\times N$ matrix of cosine similarities between every image and every caption. The $N$ diagonal entries are the *true* pairs; the $N^2 - N$ off-diagonal entries are mismatches. Training does one thing: make each diagonal entry the largest in its row *and* in its column. Do that across millions of batches and the image encoder learns to put each picture where its description's words already are.

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-3"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-3 --> **The scaled similarity.** Write $f_v$ for the image encoder and $f_t$ for the (separate, small) text encoder, with linear projections $W_I, W_T$ into the joint space. For image $I_i$ and caption $T_j$ define

<a id="eq-3"></a><!-- eq:2-3 -->
$$
s_{ij} = \frac{\mathrm{sim}(\mathbf{u}_i, \mathbf{v}_j)}{\tau}, \qquad \mathbf{u}_i = W_I f_v(I_i),\quad \mathbf{v}_j = W_T f_t(T_j),\quad \mathrm{sim}(\mathbf{a},\mathbf{b}) = \frac{\mathbf{a}^\top\mathbf{b}}{\lVert\mathbf{a}\rVert\,\lVert\mathbf{b}\rVert} \tag{3}
$$

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-4"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-4 --> where $\tau > 0$ is the temperature. Note one CLIP-specific detail that matters in practice: rather than a fixed $\tau$, CLIP learns a *log-parameterized* inverse temperature $\exp(t')$ that multiplies the cosine similarities (so $\exp(t') = 1/\tau$), initialized so that $\tau\approx 0.07$ and clipped so the multiplier never exceeds $100$ — the clip is a numerical-stability guard against the logits, and hence the softmax, blowing up <!-- cite:1 -->[[1]](#ref-1).

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-5"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-5 --> **The symmetric objective.** CLIP's loss treats each row as a $1$-of-$N$ classification problem ("which caption goes with image $i$?") and each column as the mirror problem ("which image goes with caption $j$?"), and averages the two cross-entropies — the symmetric **InfoNCE** / multi-class $N$-pair loss:

<a id="eq-4"></a><!-- eq:2-4 -->
$$
\mathcal{L}_{\mathrm{CLIP}} = -\frac{1}{2N}\sum_{i=1}^{N}\left[\, \log\frac{\exp(s_{ii})}{\sum_{j=1}^{N}\exp(s_{ij})} \;+\; \log\frac{\exp(s_{ii})}{\sum_{j=1}^{N}\exp(s_{ji})} \,\right] \tag{4}
$$

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-6"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-6 --> The first term normalizes each image's similarities over all captions (the image-to-text direction); the second normalizes each caption's similarities over all images (text-to-image). Equation <!-- ref:2-4 -->[(4)](#eq-4) is exactly CLIP's eight-line training pseudocode written as mathematics: encode, project, normalize, scale by the temperature, then a symmetric cross-entropy whose labels are the indices $0,\dots,N-1$ <!-- cite:1 -->[[1]](#ref-1). The full reading of this loss as a lower bound on the mutual information between the image and text views is developed in Appendix B; here we keep it concrete.

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-7"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-7 --> **Worked example — what the temperature does.** Take a tiny batch, $N = 4$, and suppose training has already pushed the diagonal cosine similarities to $0.40$ and left the off-diagonal ones near $0.10$. Pick the first row. With $\tau = 0.07$ the scaled logits are $s_{11} = 0.40/0.07 \approx 5.71$ on the diagonal and $s_{1j}\approx 0.10/0.07 \approx 1.43$ for the three distractors. The image-to-text softmax probability of the correct caption is then $e^{5.71}/(e^{5.71} + 3\,e^{1.43}) = 301.9/(301.9 + 3\cdot 4.18) \approx 0.96$, so that row contributes $-\log(0.96)\approx 0.04$ to the loss. Now repeat with the temperature *removed* ($\tau = 1$): the logits are just $0.40$ and $0.10$, the probability collapses to $e^{0.40}/(e^{0.40}+3\,e^{0.10}) = 1.49/(1.49+3.32)\approx 0.31$, and the loss balloons to $-\log(0.31)\approx 1.17$. The small temperature *sharpens* the similarity gap — it is a learned gain that turns a modest cosine margin of $0.30$ into a near-certain classification, which is why a sensible $\tau$ is essential and why CLIP bothers to learn it rather than guess.

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-8"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-8 --> **Intuition and the SP analogy.** Contrastive learning is matched filtering at scale. Each caption defines a template; the loss asks the image embedding to correlate maximally with its own caption's template and minimally with everyone else's, exactly as a matched filter maximizes response to the intended waveform and suppresses interferers. The "negatives" are the other pairs in the batch, which is why batch size is a first-class hyperparameter: more in-batch negatives means a harder, more discriminative classification problem and a sharper learned geometry — the matched filter sees more interferers to reject.

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-9"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-9 --> **What it predicts, and the payoff.** The aligned geometry is directly useful with *no further training*: to classify an image zero-shot, embed the candidate class names as captions ("a photo of a {dog}") and pick the nearest. CLIP reports this lifts ImageNet zero-shot accuracy from $11.5\%$ for the prior best of this kind to $76.2\%$, matching a fully supervised ResNet-50 while using none of ImageNet's labels <!-- cite:1 -->[[1]](#ref-1). For *our* purposes the deeper consequence is structural: because the encoder's features are pre-arranged by linguistic meaning, a downstream LLM needs only a light adapter to read them — the reason almost every vision–language model in § 3 starts from a frozen CLIP-style encoder rather than a bare ImageNet one. *[Established: the zero-shot number and the loss form are read directly from the CLIP paper; the "light adapter suffices" claim is the operating thesis the architecture inventory then substantiates model by model.]*

<a id="p-24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-10"></a><!-- para:24-contrastive-alignment-teaching-vision-to-speak-the-languages-geometry-10 --> **Failure mode — the single-vector bottleneck.** CLIP aligns *one* image vector to *one* caption vector, an image-level summary. That coarseness is its limitation: a single embedding cannot encode where each object sits or how many there are, so CLIP features are weak at spatial and compositional questions and can seed object hallucination downstream (§ on evaluation). The standard fix in VLMs is to feed the LLM the *grid* of patch tokens, not just the pooled class vector, recovering spatial detail at the cost of more tokens — the same resolution-versus-tokens tension from § 2.2, now in a new guise.

### <a id="sec-2.5"></a>2.5 The sigmoid alternative: SigLIP

<a id="p-25-the-sigmoid-alternative-siglip-1"></a><!-- para:25-the-sigmoid-alternative-siglip-1 --> CLIP's loss has a hidden cost buried in the denominators of Equation <!-- ref:2-4 -->[(4)](#eq-4): each one is a sum over the *entire batch*. To compute any single pair's contribution you need every other pair's similarity, so when training is sharded across $D$ accelerators the devices must all-gather their embeddings and every device must materialize the full $N\times N$ similarity matrix. That global coupling is what makes very large batches expensive. SigLIP <!-- cite:15 -->[[15]](#ref-15) removes it with a different objective.

<a id="p-25-the-sigmoid-alternative-siglip-2"></a><!-- para:25-the-sigmoid-alternative-siglip-2 --> **The reframing.** Instead of a $1$-of-$N$ classification that needs a normalizer over the batch, treat every one of the $N^2$ image–caption cells as an *independent binary decision*: is this pair a match or not? That is a sum of per-cell sigmoid (logistic) losses with no cross-cell normalization:

<a id="eq-5"></a><!-- eq:2-5 -->
$$
\mathcal{L}_{\mathrm{sig}} = -\frac{1}{N}\sum_{i=1}^{N}\sum_{j=1}^{N} \log \sigma\!\big(\, z_{ij}\,(t\,\hat{\mathbf{u}}_i^\top\hat{\mathbf{v}}_j - b)\,\big) \tag{5}
$$

<a id="p-25-the-sigmoid-alternative-siglip-3"></a><!-- para:25-the-sigmoid-alternative-siglip-3 --> where $\hat{\mathbf{u}}_i, \hat{\mathbf{v}}_j$ are the unit-normalized image and text embeddings, $\sigma$ is the logistic sigmoid, $t = \exp(t')$ is a learned temperature, $b$ is a learned bias, and the label $z_{ij}$ is $+1$ on the matched diagonal ($i=j$) and $-1$ on every off-diagonal mismatch. Each term depends only on its own cell, so the loss factorizes: no all-gather, no full similarity matrix in memory, and a simple "chunked" implementation in which each device scores only its local block of pairs.

<a id="p-25-the-sigmoid-alternative-siglip-4"></a><!-- para:25-the-sigmoid-alternative-siglip-4 --> **Why the bias term is there.** A batch has $N$ positives and $N^2 - N$ negatives, an extreme imbalance — at $N = 32{,}768$ there are tens of thousands of negatives for each positive. Left alone, the negatives dominate the gradient at initialization and the model wastes early training overcorrecting toward "everything is a mismatch." SigLIP initializes $t' = \log 10$ and $b = -10$ so that the initial logits sit far on the negative side, matching the prior that almost all pairs *are* mismatches, and training starts near the right operating point instead of fighting its way there <!-- cite:15 -->[[15]](#ref-15).

<a id="p-25-the-sigmoid-alternative-siglip-5"></a><!-- para:25-the-sigmoid-alternative-siglip-5 --> **Worked example — independence in action.** Take a freshly initialized model ($t = 10$, $b = -10$) and a typical off-diagonal pair whose embeddings are essentially uncorrelated, $\hat{\mathbf{u}}_i^\top\hat{\mathbf{v}}_j\approx 0$. Its logit argument is $z_{ij}(t\cdot 0 - b) = (-1)(0 - (-10)) = -10$, so $\sigma(-10)\approx 4.5\times 10^{-5}$ and the model already says "not a pair" with near-certainty — correctly, and *without consulting any other cell*. The matched pair in the same row, once training pushes its similarity to say $0.4$, has argument $(+1)(10\cdot 0.4 - (-10)) = 14$, giving $\sigma(14)\approx 1$. Contrast this with Equation <!-- ref:2-4 -->[(4)](#eq-4), where computing the same diagonal cell's loss required summing $e^{s_{1j}}$ across the whole row. The sigmoid form pays $O(N^2)$ in arithmetic just like the softmax, but the *memory and communication* drop because the work is embarrassingly parallel over cells.

<a id="p-25-the-sigmoid-alternative-siglip-6"></a><!-- para:25-the-sigmoid-alternative-siglip-6 --> **Tradeoff and when to prefer it.** The practical headline from SigLIP is a regime distinction: the sigmoid loss is *better than softmax at small-to-moderate batch* (notably below $16$k), and the two converge as the batch grows, with both saturating beyond a reasonable batch of about $32$k <!-- cite:15 -->[[15]](#ref-15). So SigLIP's value is not a higher ceiling but a cheaper path to a good ceiling — it decouples batch size from the loss definition, letting a strong language–image encoder be trained on a handful of accelerators rather than a large cluster. For the survey's purposes the two losses produce interchangeable *kinds* of encoder — both yield language-aligned patch features an LLM can consume — and modern VLMs draw from both families; SigLIP-pretrained encoders are an increasingly common default precisely because they are cheaper to produce.

<!-- sec:2.6 -->
### <a id="sec-2.6"></a>2.6 From encoder features to LLM tokens: the connector

<a id="p-26-from-encoder-features-to-llm-tokens-the-connector-1"></a><!-- para:26-from-encoder-features-to-llm-tokens-the-connector-1 --> We now have, from § 2.2–2.5, a sequence of $N_v$ language-aligned patch features $f_v(x_v)\in\mathbb{R}^{N_v\times d_v}$. The LLM, however, consumes vectors of *its* width $d$ in *its* embedding geometry, and in general $d_v \neq d$ and the two spaces were trained separately. The last fundamental piece is the **connector** (or projector) $g_\phi$ that maps encoder features into the LLM's token space, producing vectors that stand in the input sequence exactly where text-token embeddings would:

<a id="eq-6"></a><!-- eq:2-6 -->
$$
\mathbf{h}_i = g_\phi\big(f_v(x_v)_i\big) \in \mathbb{R}^{d}, \qquad \text{LLM input} = \big[\, \mathbf{h}_1,\dots,\mathbf{h}_{N_v},\ \mathbf{e}(x_t^1),\dots,\mathbf{e}(x_t^m)\,\big] \tag{6}
$$

<a id="p-26-from-encoder-features-to-llm-tokens-the-connector-2"></a><!-- para:26-from-encoder-features-to-llm-tokens-the-connector-2 --> where $\mathbf{e}(\cdot)$ is the LLM's ordinary text-embedding lookup and $x_t^1,\dots,x_t^m$ are the prompt's text tokens. The visual vectors $\mathbf{h}_i$ are simply *prepended to the text tokens* and the unchanged LLM processes the concatenation — this is the "early-fusion" pattern from § 1, and Equation <!-- ref:2-6 -->[(6)](#eq-6) is its mathematical core.

<a id="p-26-from-encoder-features-to-llm-tokens-the-connector-3"></a><!-- para:26-from-encoder-features-to-llm-tokens-the-connector-3 --> **The simplest connector works.** The striking empirical fact, established by LLaVA <!-- cite:5 -->[[5]](#ref-5), is that $g_\phi$ can be a *single linear layer* — one matrix $W\in\mathbb{R}^{d_v\times d}$ — and a frozen CLIP encoder plus a frozen LLM, joined by that one projection and tuned on a modest set of instruction examples, already yields a capable visual assistant. Intuitively this is the dividend of § 2.4: because the encoder's features are *already* arranged by linguistic meaning, the connector's job is mostly a change of basis into the LLM's coordinates, not a hard re-encoding of visual content. LLaVA-1.5 later swapped the single linear layer for a small two-layer MLP for a measurable bump <!-- cite:6 -->[[6]](#ref-6) — a reminder that "simplest" is the floor, not the ceiling.

<a id="p-26-from-encoder-features-to-llm-tokens-the-connector-4"></a><!-- para:26-from-encoder-features-to-llm-tokens-the-connector-4 --> **Why the connector is a whole design axis.** Equation <!-- ref:2-6 -->[(6)](#eq-6) also exposes the connector's hidden cost: it emits $N_v$ tokens, and every one of them occupies a slot in the LLM's context and pays the LLM's per-token compute *at every generated step*. A linear projector is token-preserving — $256$ patches in, $256$ visual tokens out — so a single high-resolution image can consume more context than a long paragraph of text. This is why an entire family of connectors exists not to *translate* features but to *compress* them: BLIP-2's Q-Former and Flamingo's Perceiver Resampler use a small set of learned query vectors to cross-attend to the patch features and emit a *fixed, small* number of tokens regardless of $N_v$, trading some visual fidelity for a bounded token budget. Those mechanisms — query-based resamplers, pooling connectors, deep cross-attention versus shallow projection — are the substance of the architecture inventory (§ 3) and are derived in full in Appendix C. The point to carry forward from the fundamentals is that the connector is where two independently trained networks are stitched into one, and that *how many tokens it emits* is as consequential a choice as *what geometry it maps into*.

<a id="p-26-from-encoder-features-to-llm-tokens-the-connector-5"></a><!-- para:26-from-encoder-features-to-llm-tokens-the-connector-5 --> **Closing the loop.** The four stages assembled here — partition the image into patches (§ 2.2), summarize them with an attention encoder (§ 2.3), align that encoder to language so its features are meaningful (§ 2.4–2.5), and project the aligned features into the LLM's token space (§ 2.6) — are the complete perception pathway shared, with variations, by nearly every model in this survey. With this scaffold in place we can turn to the architecture inventory, where each stage is realized in competing ways, and later to generation, where the pathway is *run in reverse* so the model can emit a modality as well as read one.
